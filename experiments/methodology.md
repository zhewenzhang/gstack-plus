# 實驗方法論

> 定義 3 種模式的公平對比規則、測量指標、和實驗局限性。
>
> **核心原則**：相同的任務、相同的起點、相同的成功標準——唯一變量是工作流模式。

---

## Part 1：3 種實驗模式定義

### Mode A：單一 Claude（基準線）

- **操作方式**：Claude 獨立完成全部工作（規劃 + 執行 + 驗證），不使用 gstack-plus 模板
- **不允許**：使用任何 Phase 1-2 創建的模板（plan-to-exec.md、exec-to-check.md 等）、使用分類器
- **角色**：Claude 既是 Architect 也是 Executor，用自然語言描述計劃、直接寫代碼、自己驗證
- **記錄方式**：手動計數步驟數和用戶介入次數，不填寫任何模板

### Mode B：手動 Claude + Qwen（無模板協作）

- **操作方式**：Claude 規劃任務，Qwen 執行，但溝通方式是自由文本（不用標準模板）
- **不允許**：使用 plan-to-exec.md、exec-to-check.md、pre-flight-checklist.md 等模板
- **角色**：Claude 寫任意格式的任務描述（自然語言，無結構要求），Qwen 執行，Claude 口頭確認結果
- **溝通特點**：Claude 給 Qwen 的指令可能是「在 src/xxx.ts 添加 xxx 功能，確保測試通過」——沒有成功標準字段、沒有 Scope Lock、沒有禁用詞掃描

### Mode C：gstack-plus（使用完整模板系統）

- **操作方式**：嚴格按照 Phase 1-2 的完整流程
  1. 用 `classifier/scoring-guide.md` 對任務評分 → `routing-rules.md` 決定路由
  2. 填寫 `verification/pre-flight-checklist.md`（12 項全部通過才能派發）
  3. 填寫 `handoff/templates/plan-to-exec.md`（任務描述、成功標準、Scope Lock、上下文快照）
  4. 執行（按路由到 Tier-Exec 或 Tier-A）並填寫 `handoff/templates/exec-to-check.md`（包含 evidence）
  5. 審查 evidence，如有失敗用 `verification/failure-routing.md` 分類和回流
- **不允許**：跳過任何步驟，或用口頭描述代替模板填寫
- **特點**：每一步都有格式要求，每個決策都有記錄

## Part 2：5 個測量指標定義

| 指標 | 定義 | 記錄方式 | 對比基準 |
|------|------|---------|---------|
| **Token 消耗** | 該 session 中所有模型調用的 input + output token 總和 | 從 IDE 統計或 API 賬單讀取。如無法獲取精確值，估算（每個對話輪次 × 平均 token 數） | Mode A 為基準 100%，C/A 和 B/A 的比值 |
| **步驟數** | 完成任務需要的「輪次」數（每次人類發送指令算 1 輪，模型返回並再次發送算第 2 輪） | 手動計數。從第一個指令開始，到成功標準全部驗證通過為止 | Mode A 為基準 |
| **回流次數** | 因錯誤需要重做或修正的次數（包括：Exec 重試、Tier-Mid 分析、Tier-A 重規劃） | 手動記錄。每次「完成但需要返工」算 1 次 | 越低越好 |
| **代碼質量** | 完成後用 gstack 的 `/review` 或同等 review 工具評分（0-100 分制） | 完成後用 `/review` 評分；如無工具，人工按代碼規範評分 | 三種模式應該接近，差距 > 10 分需要說明原因 |
| **用戶介入次數** | 需要人類做判斷/決策的次數（不含起始指令） | 手動計數。包括：確認方向、選擇方案、批准修改、回答 AI 的問題 | 越低越好 |

**Token 消耗估算指南**（如無法獲取精確數值）：
- 每輪 Claude 對話：約 2,000-5,000 input tokens + 1,000-3,000 output tokens
- 每輪 Qwen 對話：約 1,000-3,000 input tokens + 500-2,000 output tokens
- Mode C 的模板填寫：每填寫一個模板約消耗 1-2 輪對話

## Part 3：公平性規則

### 規則 1：相同輸入

3 種模式使用**完全相同**的任務描述（來自 `experiments/task-definitions.md`）。Mode B 和 Mode C 可以對任務描述做額外的結構化，但不能減少或改變任務要求。

### 規則 2：相同環境

所有模式從同一個 git commit 開始。每個模式執行完後，用 `git stash` 保存結果，恢復到初始 commit，再跑下一種模式。確保沒有模式因為「代碼已經被前一個模式改善了」而受益。

### 規則 3：成功標準統一

只有通過 `task-definitions.md` 中定義的成功標準，才算「完成」。Mode A 不能因為「看起來完成了」就結束——必須通過所有成功標準。

### 規則 4：記錄順序

先跑 Mode A（建立基準），再跑 Mode B，最後 Mode C。這樣確保：
- Mode A 的代碼質量不受其他模式影響
- Mode B 和 Mode C 的對比更公平（都有 Mode A 的結果作為參考）

### 規則 5：不干預

每種模式按照它的規則走，不在執行中途切換方式。如果 Mode C 在執行中發現模板有問題，記錄下來但不改變流程——這些觀察寫入「執行後反思」。

## Part 4：預期偏差和局限性

**誠實說明這個實驗的局限**（比較結果不能過度推廣）：

1. **樣本量小**：3 個任務、3 種模式 = 9 組數據。結論是指示性的（"gstack-plus 在這 3 個任務上表現如何"），不是統計顯著的（"gstack-plus 在所有場景都更好"）。

2. **模板填寫開銷**：Mode C 的 pre-flight checklist 和模板填寫本身消耗步驟數和 token。這部分開銷是「一次性投資」——在第一個任務中成本高，但在同類任務重複執行時會攤薄。實驗數據需要說明哪部分是模板開銷、哪部分是執行成本。

3. **設計者優勢**：Qwen 執行 Mode C 的任務時，可能受益於「設計這套模板的人（Claude）也參與了執行」——設計者更了解模板的意圖。真實場景中，Exec 可能不是設計者。

4. **Qwen 不確定性**：Mode C 使用 Qwen 作為 Exec，但 Qwen 的表現波動可能大於 Claude。如果 Qwen 在某個任務上表現特別差，可能是模型問題，不是模板問題。

5. **Mode A/B 由用戶執行**：Mode C 的數據由 Qwen Code 直接記錄，Mode A/B 的數據由用戶在獨立會話中執行後填寫。兩者的記錄精度可能不同。

6. **代碼質量評分主觀性**：`/review` 工具的評分本身有一定主觀性。對比應該關注「是否有明顯差距」（> 10 分），而不是「誰高 1-2 分」。
