# 🔍 YC 式盲點清單

> YC 教給創業者的核心理念：**「Make something people want.」**
>
> 這份文檔列出你在規劃 gstack-plus 時可能沒想到、但會決定成敗的事情。
> 不是要嚇你，是要你帶著清醒的頭腦進入。

---

## 🚨 致命級盲點（不解決就別開始）

### 1. 你和真實用戶聊過了嗎？

**YC 名言**：「Don't build what you want to build. Build what people want.」

你現在的方法論是基於**你自己**在 Louise 項目的實踐。但：

- 其他開發者也面臨同樣的痛點嗎？
- 他們已經有解決方案了嗎？
- 他們願意花時間學新工具嗎？

**行動建議**：
- [ ] 在動手前，找 5-10 個 gstack 用戶問三個問題：
  1. 你最頭疼的 AI 工作流問題是什麼？
  2. 你用過多模型協作嗎？怎麼用的？
  3. 如果有個工具能自動分配任務給不同模型，你會用嗎？為什麼？
- [ ] 在 Twitter/X、Reddit r/ClaudeAI、知乎 AI 話題下發問

**如果用戶不疼**，整個方法論的價值要重新評估。

---

### 2. 已經存在的競品你研究了嗎？

下面這些工具都在做類似的事情：

| 工具 | 重點 | 可能的差異化 |
|------|------|------------|
| **AutoGen**（微軟） | 多 Agent 對話 | 偏研究、難用 |
| **CrewAI** | Agent 團隊角色 | 偏 Python，不是 IDE 工作流 |
| **LangGraph** | 狀態機式 workflow | 偏底層，門檻高 |
| **MetaGPT** | 模擬軟件公司 | 全自動，可控性差 |
| **Aider + Cursor** | IDE 多模型 | 沒有 skill 層 |
| **OpenHands** | AI 開發者 | 端到端，不是輔助 |

**行動建議**：
- [ ] 至少裝 3 個競品，跑一遍同樣任務
- [ ] 寫一篇對比文章：`docs/competitive-analysis.md`
- [ ] 找出**你獨有的角度**（不是「我也能做」，是「他們做不了」）

**沒有差異化，就是又一個被遺忘的工具**。

---

### 3. 命名是品牌，不要將就

`gstack-plus` 有兩個問題：

**問題 A：法律風險**
- "gstack" 是 Garry Tan 的項目
- "gstack-plus" 暗示是官方延伸
- 可能引起商標/品牌爭議

**問題 B：缺乏記憶點**
- "Cursor"、"Cline"、"Aider" 都是好名字
- "gstack-plus" 像是個 fork

**行動建議**：
- [ ] **發郵件給 Garry Tan**，說明你的計劃，徵求意見
- [ ] 想 5 個原創名字（建議方向）：
  - 強調「層級」：`Layered`、`Tier`、`Cascade`
  - 強調「協作」：`Quorum`、`Council`、`Orchestra`
  - 強調「分工」：`Forge`、`Atelier`、`Studio`
- [ ] 查 GitHub、npm、商標數據庫
- [ ] 買一個短域名

**好名字省 6 個月推廣**。

## 🔥 重要級盲點

### 4. 開源不等於用戶自動上門

開源項目的 90/10 法則：**10% 是寫代碼，90% 是讓人用**。

成功的開源項目都做了：
- 引人注目的 README（圖、demo、benchmark）
- 60 秒的 demo 視頻
- HackerNews/Reddit 發文
- 找 KOL 推薦
- 持續更新和回應 issue

**行動建議**：
- [ ] 在 Phase 4 開始**前**就準備好：
  - 60 秒 demo 視頻
  - 一張 hero image（社交媒體分享用）
  - HackerNews 發布文案
- [ ] 列出 10 個目標社區/媒體
- [ ] 找 3-5 個 alpha 用戶（最好是有影響力的）

**項目最好的廣告是真實用戶的成功案例**。

---

### 5. 第一次體驗決定一切

開發者工具的殘酷真相：**用戶第一次裝失敗，永遠不再回來**。

- npm install 失敗？走人。
- 文檔跑不通？走人。
- 第一個 demo 不驚艷？忘了。

**行動建議**：
- [ ] 寫一個「60 秒 wow moment」demo
- [ ] 找 3 個朋友從 0 開始裝你的項目，**全程錄屏**
- [ ] 修掉所有他們卡住的地方
- [ ] 文檔測試：刪掉腦子裡的所有上下文，按文檔做一遍

---

### 6. 文檔就是產品（對 dev tools 來說）

看看 Stripe、Vercel、Cursor 的文檔，再看看大部分開源項目的 README。

**頂級文檔的特徵：**
- 5 分鐘看懂這是什麼
- 30 秒安裝成功
- 第一個 example 必須驚艷
- 失敗時有清晰的排錯指南
- 進階文檔結構清晰

**行動建議**：
- [ ] 給自己定下標準：**文檔質量 = Stripe 級別**
- [ ] 不要等代碼寫完才寫文檔，文檔先行
- [ ] 找一個「文檔審查員」（朋友、社區）

---

### 7. 可持續性：你會堅持下去嗎？

OSS 項目最常見的死法：**作者燒乾了**。

問自己：
- 6 個月後你還有熱情維護嗎？
- 12 個月後呢？
- 如果項目火了，你願意全職做嗎？
- 如果沒火，你願意繼續寫文檔回 issue 嗎？

**行動建議**：
- [ ] 設定一個「最低承諾」：每週至少 X 小時
- [ ] 預留時間給：回 issue、寫博客、和用戶交流
- [ ] 找 1-2 個「義務 maintainer」分擔
- [ ] 考慮**贊助/付費版本**的可能性（不是現在，是未來）

---

### 8. 度量標準先定義

YC 教導：**沒有指標，沒有進步**。

你怎麼知道項目成功了？

候選 North Star 指標：
- GitHub stars（容易，但表面）
- 月活用戶（更實在）
- 解決的真實問題數（最有意義）
- 用戶節省的成本（最 sellable）

**行動建議**：
- [ ] 在 README 第一天就埋 telemetry（匿名、可選）
- [ ] 定義 1-2 個 North Star
- [ ] 每月寫一份增長報告（公開）

## 🎓 戰略級盲點

### 9. 「AI for AI」的護城河問題

**冷酷事實**：包裝其他模型的工具，3 個月會被 OpenAI/Anthropic 自己做掉。

GPT-4 出來後死了多少 prompt 工具？Claude Sonnet 4 後又死了多少？

**你的護城河可能是：**
- ❌ 多模型路由（巨頭會自己做）
- ❌ Prompt 模板（容易被複製）
- ✅ **方法論（思考方式）** — 難以被軟件替代
- ✅ **真實案例庫** — 累積的實戰數據
- ✅ **社區** — 信任和關係
- ✅ **教育內容** — 用戶因你而成長

**行動建議**：
- [ ] 把重心從「工具」轉向「方法論 + 社區」
- [ ] 工具是免費的，方法論才是核心 IP
- [ ] 像 Garry Tan 寫博客那樣**持續產出思想**

---

### 10. 雙語策略：英文還是中文優先？

殘酷事實：
- 中文社區規模小、付費意願低、傳播範圍有限
- 英文社區規模大、影響力大、但競爭激烈

**選擇：**
- **英文優先**：對標全球 OSS 標準，難度高但天花板高
- **中文優先**：先在華語區建立口碑，再翻譯
- **雙語並行**：質量都可能受影響

**我的建議**：英文優先，中文跟進。但 README 第一版必須英文。

**行動建議**：
- [ ] 找一個英文寫作好的朋友當 reviewer
- [ ] 學 Stripe/Vercel 的英文文檔風格
- [ ] 中文版作為附帶資產

---

### 11. 法律和合規

容易忽視的：

- **License 選擇**：MIT 寬鬆但允許閉源 fork；Apache 2.0 包含專利條款；GPL 強制開源
- **依賴 license 兼容性**：你用了什麼 npm 包？它們的 license 是否兼容？
- **隱私聲明**：用戶的代碼會被發送到雲端模型嗎？必須說明
- **商標**：項目名是否被佔用？是否侵犯 gstack 商標？

**行動建議**：
- [ ] 諮詢一下：你的公司（如有）是否禁止個人開源？
- [ ] 寫 PRIVACY.md 和 TERMS.md
- [ ] 用 [choosealicense.com](https://choosealicense.com) 選 license

---

### 12. 「Talk to your users」是無止境的

YC 反復強調的點：**創業者花太少時間和用戶聊**。

不只是發布前要聊，發布後也要聊。

**行動建議**：
- [ ] 每週至少和 1 個用戶 30 分鐘通話
- [ ] 在 Discord/Slack 設立 office hours
- [ ] 每個月寫一份「用戶反饋總結」

## 🌟 加分級洞察

### 13. 跟 gstack 作者建立關係

Garry Tan 是 YC 總裁。如果他公開支持你，項目會成功 10 倍。

**怎麼做**：
- 不要冷郵件問「你能推薦我的項目嗎」
- **先貢獻**：給 gstack 提 PR、寫使用案例、在社區活躍
- 等他注意到你之後再聊合作

---

### 14. 設想 1 年後的場景

YC 的「Pre-mortem」技巧：

**想像 1 年後 gstack-plus 失敗了，原因是什麼？**

可能的原因：
- ⚠️ 用戶覺得太複雜，不如直接用 Claude
- ⚠️ 模型差距縮小，多 Tier 沒必要了
- ⚠️ 微軟/Anthropic 自己做了類似的
- ⚠️ 我自己燒乾了，沒繼續維護
- ⚠️ 沒有有意義的 case study

針對每個失敗原因，**現在就想對策**。

---

### 15. 不要害怕被別人「抄」

很多人擔心開源後被抄襲。但事實是：

- **大公司不在乎你**（你還太小）
- **小團隊抄了反而是免費宣傳**
- **執行力、社區、信任無法被抄**

**心態調整**：開源是「邀請大家來幫你變強」，不是「展示能被偷的寶藏」。

## 💎 一句話總結

**YC 哲學**：

> 「The best founders are obsessed with talking to users,
> learning from them, and shipping fast.」
>
> 「Make something people want.」

回到根本：

- 不是「我能做這個工具」
- 是「有人需要這個工具」
- 你能否做到「他們需要的更多功能」
- 你能否堅持到「他們真的離不開你」

**做之前，請反復問自己這些問題**。
