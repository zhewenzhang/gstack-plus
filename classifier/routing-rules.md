# 路由決策樹

> 任務分類器的第二部分：拿到 5 個維度的評分後，用這個決策樹路由到正確的 Tier。
>
> **保守路由原則**：不確定時，默認路由到更高 Tier。低估任務複雜度的代價，遠大於高估。

---

## 路由規則

### 規則 1：Tier-A（戰略決策層）

```
判斷強度 ≥ 4
  OR 風險權重 ≥ 4
  OR 創意密度 ≥ 4
→ Tier-A
```

**理由**：這些維度的高分意味著任務需要架構級別的判斷、涉及高風險、或需要從零設計。這些是 Tier-Exec 和 Tier-Mid 不擅長的事。

---

### 規則 2：Tier-Exec（執行層）

```
判斷強度 ≤ 2
  AND 上下文寬度 ≤ 2
  AND 可驗證性 ≥ 4
→ Tier-Exec
```

**理由**：這個任務是機械性的（判斷低）、局部性的（上下文窄）、可驗證的（有明確成功標準）。完美適合 Exec 執行。

**注意**：可驗證性 ≥ 4 是硬性要求。如果任務不可驗證（可驗 ≤ 3），即使判斷和上下文都很低，也不適合 Exec——因為無法自動證明 Exec 做對了。

---

### 規則 3：Tier-Mid（複雜檢查層）

```
不滿足規則 1 且不滿足規則 2
→ Tier-Mid
```

**理由**：這個任務需要一些判斷力，但不需要架構級別的推理；或者它涉及較多上下文，但風險不高。這是 Tier-Mid 的甜蜜區。

## 特殊情況處理

### 情況 1：某個維度缺乏信息

**問題**：評分某個維度時，發現信息不足（例如：不知道這個任務涉及多少文件）。

**處理**：**預設值 → 給高分（保守路由）**

```
信息不足的維度 → 按高分評分 → 套用路由規則
```

**理由**：低估任務複雜度的代價（Exec 搞砸了需要從零重來）遠大於高估（用了一個比需要更強的模型）。

**示例**：
- 「這個任務需要改多少文件？」→ 不知道 → 上下文寬度給 4 → 路由到 Tier-Mid 或 Tier-A
- 「這個任務做錯了有什麼風險？」→ 不知道 → 風險權重給 4 → 路由到 Tier-A

---

### 情況 2：Tier-Mid 不可用時的降級方案

**問題**：Tier-Mid 模型暫時不可用（API 限流、維護等）。

**降級方案**：
- 原應路由到 Tier-Mid 的任務 → **升級到 Tier-A**
- 如果 Tier-A 也不可用 → **延遲執行，排隊等待**

**不建議的降級**：
- ❌ 降級到 Tier-Exec：Mid 的任務通常需要判斷力，Exec 搞砸的概率高

**理由**：Tier-A 比 Tier-Mid 貴，但比 Tier-Exec 搞錯後返工便宜。

---

### 情況 3：任務描述本身有歧義

**問題**：任務描述不夠清晰，分類器無法可靠評分。

**處理**：**升 Tier-A 澄清，不猜測**

```
任務描述有歧義 → Tier-A 澄清 → 重新評分 → 路由
```

**理由**：基於歧義描述的評分是不可靠的，路由結果也靠不住。讓 Tier-A（有架構視角）來澄清是最安全的選擇。

**示例**：
- 「優化性能」→ 優化哪裡？→ 歧義 → Tier-A 澄清
- 「更新文檔」→ 哪些文檔？→ 歧義 → Tier-A 澄清

## 邊界案例指南

### 看起來簡單但其實需要 Tier-A 的任務特徵

| 特徵 | 例子 | 為什麼需要 Tier-A |
|------|------|-----------------|
| **隱含架構影響** | 「把這個函數改成異步的」 | 所有調用者都需要同步修改，影響鏈不明顯 |
| **隱含安全風險** | 「在 URL 中傳遞用戶 ID」 | 可能導致 IDOR 漏洞，但表面看起來只是格式修改 |
| **隱含數據風險** | 「把這個字段從 string 改成 number」 | 現有數據需要遷移，可能丟失精度 |
| **看起來小但依賴深** | 「改一個常量值」 | 這個常量被 50 個地方引用，改變行為 |

**識別技巧**：問「這個改動的影響鏈有多深？」如果影響鏈超過 2 層，可能是偽簡單任務。

---

### 看起來複雜但可以 Tier-Exec 的任務特徵

| 特徵 | 例子 | 為什麼適合 Exec |
|------|------|----------------|
| **看起來大但是機械的** | 「把所有文件的 license header 更新為 2026」 | 涉及 100+ 文件，但操作完全一致 |
| **有完整模板參考** | 「按照 auth 模塊的模式，給 billing 模塊加同樣的錯誤處理」 | 複雜但有模板，Exec 可以照搬 |
| **可驗證性極高** | 「跑 benchmark，如果 p99 > 100ms 就報警」 | 有明確的通過/失敗標準 |

**識別技巧**：問「這個任務能寫一個腳本自動完成嗎？」如果能，Exec 可以替代。

## 保守路由原則

> **不確定時，默認路由到更高 Tier。低估任務複雜度的代價，遠大於高估。**

**具體操作**：

1. **評分時不確定**（該給 3 還是 4？）→ **給 4**
2. **路由結果在邊界**（剛剛好不滿足 Exec 條件）→ **升一級到 Tier-Mid**
3. **任務類型是第一次出現**（以前沒有做過同類任務）→ **升一級**
4. **任務涉及用戶數據或安全**→ **直接 Tier-A**

**理由**：
- 高估的代價：多用了一些高階模型的 token，成本高 20-50%
- 低估的代價：Exec 搞砸了 → Tier-Mid 分析 → Tier-A 重新規劃 → 總成本是原來的 3-5 倍

**長期優化**：隨著經驗積累，同類任務的評分會趨於穩定。當某類任務連續 5 次都由 Exec 成功完成時，可以降低 1 分（從 3 降到 2），讓路由更激進。
