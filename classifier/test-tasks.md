### 任務 19：代碼審查：`src/auth/login.ts` 中的密碼驗證邏輯

**任務描述**：審查 `src/auth/login.ts` 中的 `validatePassword` 函數。檢查：密碼強度要求是否符合 OWASP 標準（最小長度、大小寫、數字、特殊字符）、是否有暴力破解保護（rate limiting）、錯誤消息是否洩漏用戶是否存在的信息。

**正確路由**：Tier-A

**評分**：
| 維度 | 分數 |
|------|------|
| 判斷強度 | 4 |
| 上下文寬度 | 2 |
| 風險權重 | 4 |
| 可驗證性 | 3 |
| 創意密度 | 2 |

**路由推導**：
- 判斷強度 = 4 → 觸發 Tier-A 條件（判斷 ≥ 4）
- 風險權重 = 4 → 觸發 Tier-A 條件（風險 ≥ 4）
- 兩個獨立條件都指向 Tier-A，路由結果明確。

**保守路由原則示範**：
這個任務有一個誤導性特徵——有現成的 OWASP checklist 可以參考，讓人覺得
「按 checklist 照做就好，不需要 Tier-A 的判斷」。

這個想法是錯的。OWASP checklist 告訴你「要檢查什麼」，但不告訴你「在這個
具體代碼庫裡，第 3 條的風險等級高不高、怎麼修」。後者需要對代碼架構的理解，
這正是判斷強度=4 的原因。

保守路由原則：有 checklist 只是降低了知識門檻，不降低判斷需求。
安全任務（風險=4）的代價是不對稱的——低估給 Tier-Exec 做錯的代價，
遠大於高估給 Tier-A 多花的 token 成本。

## 邊界案例（任務 20-21）

### 任務 20：將項目的錯誤處理從 callback 風格改為 async/await

**任務描述**：項目中約有 40 個函數使用 callback 風格的錯誤處理（`function(err, result)`）。將它們全部改為 `async/await` 風格。調用者也需要同步修改。現有測試應該全部通過。

**正確路由**：Tier-Mid

**評分**：
| 維度 | 分數 |
|------|------|
| 判斷強度 | 3 |
| 上下文寬度 | 4 |
| 風險權重 | 3 |
| 可驗證性 | 4 |
| 創意密度 | 1 |

**路由推導**：上下文 = 4, 判斷 = 3, 風險 = 3, 可驗 = 4。不滿足 Exec（上下文 > 2, 判斷 > 2），不滿足 Tier-A（判斷 < 4, 風險 < 4, 創意 < 4）→ Tier-Mid

**反直覺點**：這個任務看起來是「機械操作」（改語法），但上下文寬度大（40 個函數 + 它們的調用者），而且 callback→async 的改動可能改變錯誤傳播行為（需要判斷每個改動是否安全）。所以它不是純 Exec。

**容易判斷錯的原因**：只看操作類型（語法轉換）會以為是 Exec，但實際需要理解每個函數的錯誤傳播路徑。

**任務描述**：添加 `GET /health` 端點，返回 JSON：`{ status: "ok", uptime: number, timestamp: string }`。需要檢查：這個端點是否應該在認證 middleware 之前執行（應該是，健康檢查不需要認證）、是否應該記錄日誌（應該，但用 debug 級別）、是否需要 rate limiting（不需要，因為返回的信息很少且簡單）。

**正確路由**：Tier-Exec

**評分**：
| 維度 | 分數 |
|------|------|
| 判斷強度 | 2 |
| 上下文寬度 | 2 |
| 風險權重 | 1 |
| 可驗證性 | 4 |
| 創意密度 | 1 |

**路由推導**：判斷 ≤ 2 AND 上下文 ≤ 2 AND 可驗 ≥ 4 → Tier-Exec

**反直覺點**：任務描述中包含了架構判斷的結果（「是否應該在認證 middleware 之前執行」），這些是 Tier-A 已經做出的決策。Exec 只需要按決策執行。如果任務描述是「添加 /health 端點」（沒有具體說明這些設計決策），那就需要 Tier-Mid 來做這些判斷。

**容易判斷錯的原因**：如果任務描述沒有包含設計決策，評分的判斷強度會升到 3-4，路由結果變成 Tier-Mid 或 Tier-A。關鍵在於任務描述是否已經包含了決策。
