# 失敗回流報告

> Tier-Mid（Reviewer）→ Tier-A（Architect）的回流格式。
>
> **用途**：當 Exec 的產出經審查後發現不合格，或任務本身分解有問題時，使用此模板回流給 Tier-A。
>
> **核心思維**：失敗不是 Exec 的錯——可能是任務分解太難、上下文不足、或 Exec 的能力邊界。回流報告的目的是診斷「哪裡錯了」，不是指責「誰錯了」。

- **回流時間**：`[ISO 8601]`
- **Tier-Mid 分析者**：`[名稱/角色]`

---

## 回流類型說明

**BUILD_ERROR**：
編譯/測試失敗。可能 Exec 重試 1 次可解決。
典型場景：語法錯誤、類型不匹配、依賴缺失。

**LOGIC_ERROR**：
代碼跑通了，但邏輯不符合成功標準。需 Tier-Mid 分析根因。
典型場景：函數返回值錯誤、邊界情況未處理、業務邏輯錯誤。

**DESIGN_ISSUE**：
任務本身的分解有問題，Exec 無法在當前邊界內完成。需 Tier-A 重新規劃。
典型場景：任務要求修改 A 但實際上需要先修改 B、任務依賴不存在的接口。

**SCOPE_DRIFT**：
完成任務需要超出 Scope Lock 的修改。需 Tier-A 評估是否擴大範圍。
典型場景：實現功能需要新增工具函數、修改配置文件、更新依賴版本。

## 失敗 Evidence

```yaml
failure_evidence:
  attempted_at: "[時間]"
  error_type: "[BUILD_ERROR / LOGIC_ERROR / DESIGN_ISSUE / SCOPE_DRIFT]"
  error_detail: "[具體錯誤信息或失敗的成功標準。如 BUILD_ERROR，貼完整編譯錯誤；如 LOGIC_ERROR，貼測試失敗輸出]"
  exec_attempts: 1   # Exec 嘗試次數

  what_was_tried:
    - attempt: 1
      action: "[Exec 嘗試了什麼]"
      result: "[結果：錯誤是否減少？]"
    - attempt: 2
      action: "[第二次嘗試（如有）]"
      result: "[結果]"

  # 如 Exec 有返回 exec-to-check.md，複製其 evidence 和 concerns 字段
```

## 使用指南

1. **Tier-Mid 發現問題**：審查 Exec 產出後，發現不合格
2. **Tier-Mid 填寫**：填寫此報告（基本信息 → 失敗 Evidence → 根因分析 → 建議行動）
3. **發送給 Tier-A**：將此報告作為回流輸入
4. **Tier-A 回應**：填寫「Tier-A 回應」字段，決定後續行動
5. **執行**：按決定行動（重試、修復、重新分解、或升級）
