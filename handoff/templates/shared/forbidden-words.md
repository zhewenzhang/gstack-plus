# 禁用詞速查卡

> 給 Tier-A（Architect）使用。填寫 `plan-to-exec.md` 之前，掃描任務描述。
> 如果出現以下詞語，必須替換掉再發送。
>
> **核心原則**：任務描述必須讓一個「熟悉技術但不了解這個項目」的開發者能直接執行。

---

## 禁用詞清單

### ❌ 「適當的」

**問題**：Exec 會用自己的標準判斷什麼是「適當的」——但 Exec 沒有全局視角，它的標準可能與項目不符。

**替換思路**：改為具體的條件或標準。

```
❌ 錯誤：「添加適當的錯誤處理」
✅ 正確：「如果遇到 NetworkError，重試 2 次（間隔 1 秒），然後返回 500 狀態碼」

❌ 錯誤：「使用適當的數據結構」
✅ 正確：「使用 Map<string, User>，以 userId 為 key」

❌ 錯誤：「設置適當的超時時間」
✅ 正確：「設置超時時間為 30 秒」
```

---

### ❌ 「合理的」

**問題**：和「適當的」一樣——Exec 的「合理」可能不正確。

**替換思路**：改為可測量的標準。

```
❌ 錯誤：「確保性能合理」
✅ 正確：「確保響應時間不超過 100ms」

❌ 錯誤：「代碼結構應該合理」
✅ 正確：「通過 lint 規則檢查，cyclomatic complexity < 10」

❌ 錯誤：「返回合理的錯誤消息」
✅ 正確：「返回格式為 { error: string, code: number } 的對象」
```

---

### ❌ 「如果需要」

**問題**：Exec 通常會選擇「不需要」——因為這是最快完成任務的路徑。

**替換思路**：明確告訴 Exec「必須」還是「不需要」。

```
❌ 錯誤：「如果需要，添加日誌輸出」
✅ 正確：「必須添加日誌輸出，格式為 [INFO][timestamp][functionName] message」

❌ 錯誤：「如果需要，處理邊界情況」
✅ 正確：「必須處理以下邊界情況：null 輸入、空數組、超過 1000 個元素的數組」

❌ 錯誤：「如果需要，更新測試」
✅ 正確：「必須為新函數添加測試，覆蓋 null 輸入、正常輸入、邊界情況」
```

---

### ❌ 「根據情況」「視情況而定」

**問題**：把決策推給 Exec，但 Exec 沒有全局視角來做這個決策。

**替換思路**：在派發前由 Tier-A 做決策，明確寫入任務描述。

```
❌ 錯誤：「根據用戶角色顯示不同的內容」
✅ 正確：「admin 用戶顯示全部數據，viewer 用戶只顯示 name 和 email 字段」

❌ 錯誤：「根據環境選擇不同的數據庫配置」
✅ 正確：「development 環境使用 SQLite，production 環境使用 PostgreSQL」
```

---

### ❌ 「類似地」「以此類推」「同樣地」

**問題**：Exec 可能推斷錯誤的類比對象。每個需要修改的位置必須明確列出。

**替換思路**：明確列出所有需要修改的位置。

```
❌ 錯誤：「在 User 組件中添加 loading 狀態，其他組件類似地處理」
✅ 正確：「在以下組件中添加 loading 狀態：
          - src/components/User.tsx
          - src/components/Order.tsx
          - src/components/Product.tsx
          實現方式：添加 isLoading prop，true 時顯示 <Spinner />」

❌ 錯誤：「為 getAll 函數添加分頁，其他 CRUD 函數以此類推」
✅ 正確：「為以下函數添加分頁：
          - getAllUsers(page, limit)
          - getAllOrders(page, limit)
          - getAllProducts(page, limit)
          返回格式：{ data: T[], total: number, page: number, limit: number }」
```

## 快速自查

填寫完任務描述後，問自己這個測試問題：

> **這個描述，一個「熟悉技術但對這個項目一無所知」的開發者，能直接執行嗎？**

- **能，不需要任何額外信息** → ✅ 合格
- **需要猜測一些細節** → ❌ 不合格，補充具體條件
- **需要知道項目特有的約定** → ❌ 不合格，補充上下文快照

## 常見模式速查

| 模糊表達 | 問題 | 具體替換 |
|---------|------|---------|
| 「處理錯誤」 | 什麼錯誤？怎麼處理？ | 「如果 catch 到 TypeError，返回 400；如果 catch 到 NetworkError，重試 1 次後返回 503」 |
| 「優化性能」 | 優化哪裡？怎麼衡量？ | 「將 getUsers 的響應時間從 500ms 降到 100ms 以下，使用 Redis 緩存」 |
| 「改進代碼質量」 | 什麼算改進？ | 「使 cyclomatic complexity 從 15 降到 8 以下，通過提取函數實現」 |
| 「確保安全」 | 哪些安全問題？ | 「添加 CSRF token 驗證，防止 SQL injection（使用 parameterized query）」 |
| 「更新文檔」 | 哪些文檔？ | 「更新 README.md 的 Installation 和 Usage 章節，反映新的環境變量要求」 |
