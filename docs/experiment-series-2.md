# 實驗報告：Series 2 — 多維度有效性驗證

> **快速導航**：[← 上一系列](#) · [→ 下一系列](experiment-series-3.md) · [全系列總結](experiment-summary.md) · [6 個核心發現](key-findings.md)

> **實驗日期**：2026-05-05
> **任務規模**：30 任務（穩定性）+ 9 任務（成本）+ 9 任務（質量）
> **總 API 成本**：$0.50（含 LLM-as-Judge）

---

## 為什麼要做 Series 2

Series 1 實驗（2026-05-04）基於 3 個任務，得出「路由模型比 All-Opus 節省 46% 成本」的結論。但這個結論有兩個問題：

1. **樣本量太小**：3 個任務不足以得出統計意義上的結論
2. **質量評估靠人工**：主觀打分，評分者已知哪個是哪個模型，存在偏差

Series 2 用三個相互補充的實驗重新驗證，解決這兩個問題。

---

## 實驗設計

| 實驗 | 問題 | 方法 | 成本 |
|------|------|------|------|
| **Exp-2A 路由穩定性** | 評分 ±1 偏差會導致多少任務路由出錯？ | 30 個任務 × 全維度 ±1 擾動，純計算 | $0 |
| **Exp-2B 擴展成本基準** | 46% 節省只是巧合還是規律？ | 9 個新任務（3 per tier），All-Opus vs 路由，18 次 API 調用 | $0.49 |
| **Exp-2C LLM-as-Judge** | 路由後質量下降了多少？能客觀量化嗎？ | Claude Haiku 作盲評 judge，隨機 A/B 排序防位置偏差，三維度 1-5 分 | $0.01 |

## 核心發現

### 1. 路由算法高度可靠

**路由準確率：100%（30/30 任務）**

在 30 個覆蓋三個 Tier 的多樣化任務中，路由算法對每個任務都給出了符合預期的 Tier 分配，沒有任何「錯誤路由」。

**路由穩定性：平均 87%**

對每個任務的每個維度做 ±1 的輕微擾動（模擬評分者打分偏差），87% 的擾動不會改變路由結果。

各維度對路由的影響（改變 Tier 的頻率）：

| 維度 | 路由改變率 | 意義 |
|------|-----------|------|
| **Judgment 判斷強度** | **32%** | 最關鍵，±1 會改變 1/3 的任務路由 |
| Risk 風險權重 | 18% | 次關鍵 |
| Context 上下文寬度 | 9% | 中等影響 |
| Creativity 創意密度 | 8% | 較低影響 |
| Verifiability 可驗證性 | 6% | 影響最小 |

**實踐建議**：為 Judgment 維度投入最多的校準精力，它是路由結果的主要決定因子。

### 2. Tier-Exec 是最高 ROI 場景

**成本節省：98% | 質量：Mode B 更優（14.7 vs 13.7 /15）**

Tier-Exec 任務（機械可執行，高可驗證性）路由到 Qwen，不僅成本接近為零（$0.00013 vs $0.00630），連質量評分也高於 Opus。這個「既省錢又更好」的結果來自任務特性本身：Exec 任務有明確的正確答案，Qwen 的簡潔輸出反而比 Opus 的詳細輸出更「可操作」。

### 3. Tier-Mid 存在成本-質量 Tradeoff

**成本節省：85% | 質量：Mode A 更優（15.0 vs 12.7 /15）**

Tier-Mid 任務（需要工程判斷，中等複雜度）路由到 Sonnet，節省 85% 成本，但 LLM-as-Judge 評估顯示質量有所下降（12.7 vs 15.0 /15）。差距主要在「完整性」和「可操作性」維度——Sonnet 的回答準確但深度不足，Opus 的回答更加細緻周全。

**優化方向**：改進 Tier-Mid 任務的 prompt 設計，或在質量要求更高的 Mid 任務中考慮使用 Opus。

### 4. 整體質量持平

**Mode A：14.1/15 | Mode B：14.1/15 | 平局：3A vs 3B vs 3 ties**

跨所有 9 個任務的平均質量完全相同。即使 Tier-Mid 存在質量差距，Tier-Exec 的反向優勢（Mode B 更好）也完全補偿回來。這驗證了 gstack-plus 的核心主張：**路由不降低整體質量**。

### 5. Tier-Exec 是「更快 + 更便宜 + 更優質」的三重贏

**延遲改善：Exec -81% | Mid -20% | Tier-A ±0%**

延遲數據揭示了比成本節省更直接的價值：

| Tier | Mode A (All-Opus) 平均延遲 | Mode B (路由) 平均延遲 | 改善 |
|------|--------------------------|----------------------|------|
| **Tier-Exec** | 8,778 ms | **1,708 ms** | **-81%** |
| **Tier-Mid** | 11,087 ms | **8,860 ms** | -20% |
| Tier-A | 19,860 ms | 21,927 ms | ≈ 持平 |

Tier-Exec 任務用 Qwen 不只便宜 98%，回應速度還快 5 倍。在一個開發者每天處理十幾個 Exec 級別任務的場景下，累積的延遲節省直接等同於開發效率提升。

**Tier-Exec 是唯一同時實現三重優勢的 Tier**：成本節省 98%、延遲減少 81%、質量反而更高（14.7 vs 13.7）。

### 6. 成本節省：27%（更真實的工作分布）

Series 2 採用均衡的任務分布（3 Exec + 3 Mid + 3 A），比 Series 1 更接近實際工作場景。在這個分布下，整體成本節省 27%（$0.495 → $0.363）。

| 工作場景 | 預期成本節省 |
|---------|-----------|
| Exec 任務為主（如 CI/CD 自動化） | 接近 98% |
| 均衡分布（真實開發工作） | **約 27-46%** |
| Tier-A 任務為主（如架構設計） | 接近 0% |

## 對方法論的反思

**LLM-as-Judge 是可靠的質量評估工具**

9 個任務的盲評全部顯示「高置信度」，且每個任務的評分差異清晰可見。成本僅 $0.01，比人工評估更快、更一致，是未來實驗的標準方法。

**邊界任務需要保守路由**

最不穩定的 5 個任務（穩定率 60-70%）全是邊界案例——它們的評分剛好處於 Tier 邊界上。建議：**當難以判斷時，向上路由到更高 Tier**。這是「保守路由原則」——超出能力範圍的錯誤代價遠高於過度使用高 Tier 的成本。

## 研究局限性

本報告的結論需要在以下背景下解讀：

1. **合成任務**：所有 9 個測試任務都是為實驗設計的，並非從真實項目中提取。真實任務的語言模糊性、業務上下文依賴、不完整規格等特征在合成任務中被簡化了。

2. **Tier-A 質量差異是隨機性，非模型差異**：Mode A 和 Mode B 的 Tier-A 任務都使用 Opus，所以 A1/A3 出現的質量差距（13/15 vs 15/15）源於 API 溫度帶來的輸出隨機性，不代表「路由到 Opus 比直接用 Opus 更好」。

3. **小樣本**：每個 Tier 只有 3 個任務，無法排除樣本偏差。Tier-Mid 的質量下降（12.7 vs 15.0）需要更多數據確認。

4. **單一 Judge 模型**：LLM-as-Judge 使用 Claude Haiku，不同 judge 模型可能給出不同評分分布。

---

## 下階段優化方向

基於實驗數據，按優先級排列：

1. **Judgment 維度校準**：±1 影響 32% 路由，是最值得投入的校準工作。在 Playground 中加入更清晰的 Judgment 分級示例。

2. **Tier-Mid Prompt 優化**：Sonnet 在 Mid 任務上質量下降（12.7 vs 15.0）。需要測試更具針對性的 prompt 設計是否能縮小差距。

3. **邊界案例識別**：在 Playground 中加入「邊界警告」——當某個任務的評分接近 Tier 邊界時，提示用戶考慮保守路由。

4. **Series 3 實驗**：在真實項目中採集 30+ 個自然產生的任務（非合成任務），測試路由準確率是否在 100% 附近保持穩定。

## Series 3 預覽

### Exp-3A：Tier-Mid Prompt 優化結果

| Prompt 策略 | Sonnet 平均分 | vs Opus | 質量差 |
|------------|-------------|---------|------|
| S0 基準（無 system prompt）| 13.7/15 | 15.0/15 | -1.3 |
| S1 角色 + 深度加強 | **15.0/15** | 12.7/15 | **+2.3** ✓ |
| S2 結構化輸出格式 | 13.3/15 | 14.0/15 | -0.7 |
| S3 思維鏈觸發 | **15.0/15** | 14.0/15 | **+1.0** ✓ |
| *Opus 基準* | *14.1/15* | — | |

**結論**：S1（角色 + 深度加強）和 S3（思維鏈觸發）兩策略讓 Sonnet 在 Mid 任務上**超越 Opus 基準**。S1 以「staff-level engineer」角色設定配合「opinionated, surface non-obvious risks」指令，是最一致的贏家。
