# 掘金文章草稿

**標題**：我為什麼不再把所有 AI 任務都丟給最貴的模型

**標籤**：AI、效率工具、Claude、LLM、工程方法論

---

## 前言

用 AI 寫代碼快一年了，我遇到一個反覆出現的問題：

**什麼任務應該用哪個模型？**

一開始我的答案是「都用最好的」——Claude Opus、GPT-4，反正能力強就往上扔。結果：
- 費用燒得很快
- 響應速度慢
- 更重要的是：**對許多任務根本沒有差別**

後來我開始思考一個問題：**AI 任務有沒有辦法像代碼架構一樣，分層處理？**

---

## 三層模型框架

我把 AI 任務分成三層：

| 層級 | 模型 | 適用場景 |
|------|------|---------|
| Tier-A | Claude Opus、GPT-4 | 架構決策、安全審查、創意設計 |
| Tier-Mid | Claude Sonnet、GPT-4o | 代碼審查、邏輯校對、複雜驗證 |
| Tier-Exec | Qwen Code、Haiku、DeepSeek | 機械執行、模式應用、文件生成 |

核心思想：**讓對的層級做對的事。**

---

## 5 維度評分系統

關鍵問題是：**怎麼判斷一個任務屬於哪層？**

我設計了 5 個維度，每個維度打 1-5 分：

### 1. 判斷強度（Judgment Strength）
這個任務需要多少推理能力？

- 1 分：修改一個 ESLint 規則（機械操作，答案是確定的）
- 5 分：設計一個認證系統的架構（需要權衡多種方案）

### 2. 上下文寬度（Context Width）
這個任務需要理解多大範圍的代碼？

- 1 分：改一個函數名（只看一個文件）
- 5 分：重構整個數據流（需要理解整個系統）

### 3. 風險權重（Risk Weight）
如果做錯了，後果有多嚴重？

- 1 分：改一個 CSS 變量（最多顯示問題）
- 5 分：修改數據庫 migration（出錯可能丟失數據）

### 4. 可驗證性（Verifiability）
能不能客觀確認任務成功？

- 1 分：「讓這段話更有說服力」（純主觀）
- 5 分：`npm run build` 通過（客觀，自動化可驗證）

### 5. 創意密度（Creativity Density）
這個任務需要多少開放式思考？

- 1 分：套模板生成文件（填空題）
- 5 分：設計 API 接口（需要從零創造）

---

## 路由規則

有了 5 個維度的分數，路由規則很簡單：

```
判斷 ≥ 4 OR 風險 ≥ 4 OR 創意 ≥ 4  →  Tier-A

判斷 ≤ 2 AND 上下文 ≤ 2 AND 可驗證 ≥ 4  →  Tier-Exec

其他  →  Tier-Mid
```

**保守原則**：不確定時往上路由。

---

## 實際效果

我用這套方法做了一個 18 個 Phase 的項目（gstack-plus CLI 工具本身）：

- Claude Sonnet 負責：規劃、審查、任務文件撰寫
- Qwen Code 負責：所有代碼修改、文件生成

結果：Tier-A（Opus）的使用降低了約 60%，而且質量沒有明顯下降。

## 工具

我把這個方法論做成了一個 CLI 工具：

```bash
npm install -g gstack-plus

# 評分一個任務並生成移交文檔
gstack-plus classify

# 瀏覽 5 個示範案例（ESLint fix / 重構 / 認證系統等）
gstack-plus examples

# 查看評分規則說明
gstack-plus rules
```

- GitHub：https://github.com/zhewenzhang/gstack-plus
- 文件網站：https://zhewenzhang.github.io/gstack-plus/
- npm：`npm install -g gstack-plus`

## 什麼情況下不適用

這個框架**不是銀彈**：

1. **任務很少時**：如果你一天只問 AI 幾個問題，不值得花時間評分
2. **任務邊界模糊時**：有些任務評分很主觀，不如直接用 Tier-A
3. **速度優先時**：評分本身也有成本，緊急情況下直接用最好的模型

---

## 結語

最貴的模型不代表最好的結果。

讓對的模型做對的任務，才能在成本和質量之間找到最佳平衡。

有問題歡迎評論，特別是「你有沒有覺得這 5 個維度有遺漏」這個問題——我也還在思考。
