# 路由信心指南

> 基於 Series 6 實驗發現：21% 的邊界任務在評分誤差 ±1 時會改變路由。本指南幫助你在評分邊界時做出正確決策。

---

## 1. 什麼是「信心路由」

每個任務的評分應該附帶信心等級：

| 信心等級 | 條件 | 行動 |
|---------|------|------|
| **高信心（High）** | 所有維度評分清晰，無歧義 | 直接按評分路由 |
| **中信心（Medium）** | 1-2 個維度有歧義，但不影響路由 | 按評分路由，記錄疑慮 |
| **低信心（Low）** | J 或 R 處於邊界值（=3 vs 4），且任務重要 | 觸發保守路由 |

---

## 2. 保守路由觸發條件

以下任一條件成立時，自動升一個 Tier：

```
條件 A：J = 3（判斷強度邊界）且任務影響 production
條件 B：R = 3（風險邊界）且無法快速回滾
條件 C：兩個或以上維度同時在邊界（如 J=3 且 C=2 且 V=4，正好在 Exec/Mid 邊界）
```

**為什麼保守路由值得**：Series 6 顯示 R 和 J 的敏感率分別為 33% 和 32%。保守路由的額外成本（約 $0.009/任務）遠低於錯誤路由的代價。

## 3. 各維度評分決策樹

### J（判斷強度）

- 任務中有幾個「沒有標準答案」的決策點？
  - 0 個 → **1-2**
  - 1-2 個 → **2-3**
  - 3 個以上 → **4-5**
- 合理的工程師之間是否可能對最佳做法有分歧？
  - 否 → ≤2
  - 可能 → 3
  - 很可能 → ≥4

**常見錯誤**：「我不確定怎麼做」不等於 J 高，是 C 高（需要更多上下文）。

### R（風險權重）

- 如果做錯了，修復成本是多少？
  - < 30 分鐘 → **1-2**
  - 數小時 → **3**
  - 數天 → **4**
  - 災難性 → **5**
- 是否涉及 prod 數據庫、用戶認證、計費系統？
  - 是 → **自動 ≥4**

**常見錯誤**：安全相關代碼的 R 往往被低估為 2，應為 4+。

### C（上下文寬度）

- 執行任務需要閱讀多少個文件？
  - 0-2 個 → **1-2**
  - 3-5 個 → **3**
  - 5 個以上 → **4-5**

### V（可驗證性）

- 任務完成後能用測試/腳本 100% 確認？
  - 能 → **4-5**
  - 部分 → **2-3**
  - 無法自動驗證 → **1-2**

**常見錯誤**：「我可以手動測試」= V=2，「CI 腳本可以驗證」= V=4。

### Cr（創意密度）

- 任務要求設計「之前不存在的東西」嗎？
  - 否 → **1-2**
  - 局部創新 → **3**
  - 全新設計 → **4-5**

## 4. 邊界案例對照表

| 場景 | 推薦評分 | 路由 | 說明 |
|------|---------|------|------|
| 新增 log 語句到指定函數 | J=2,C=2,R=1,V=5,Cr=1 | Tier-Exec | 完全機械，V=5 |
| 重構大型組件拆分 | J=3,C=3,R=3,V=3,Cr=3 | Tier-Mid（保守路由候選） | R=3 且 J=3，需評估是否升 Tier-A |
| 設計 API 限流策略 | J=4,C=2,R=3,V=4,Cr=2 | Tier-A | J=4 單維度觸發 |
| 更新 CI 環境變量 | J=2,C=2,R=2,V=3,Cr=1 | Tier-Mid | 不滿足 Exec 條件（V<4） |

---

## 5. 常見評分錯誤

基於 Series 6 發現的最常見錯誤：

1. **R 低估**：安全相關代碼的 R 往往被低估為 2，應為 4+
2. **J 高估**：「我不確定怎麼做」不等於 J 高，是 C 高（需要更多上下文）
3. **V 混淆**：「我可以手動測試」= V=2，「CI 腳本可以驗證」= V=4
