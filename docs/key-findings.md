# gstack-plus：6 個核心實驗發現

> 來自 6 個獨立實驗系列（2026-05-04 至 2026-05-06）的最重要量化結論。
> 完整報告：[實驗完整摘要](experiment-summary.md)

---

## 發現 1：路由削減 46%–98% 成本，依 Tier 結構分佈

| Tier | 典型節省 | 舉例 |
|------|---------|------|
| Tier-Exec | **−98% 至 −99%** | $0.012 → $0.00014（跨 repo 重命名） |
| Tier-Mid | **−81% 至 −86%** | $0.079 → $0.012（React Query 重構） |
| Tier-A | 0% | 架構設計仍需最強模型 |
| **整體（真實 git 任務分佈）** | **−46%** | 45% Exec + 35% Mid + 20% Tier-A |

**關鍵洞察**：節省幅度的差異揭示了一個重要事實——大多數開發任務（80%）根本不需要最昂貴的模型，而目前的工作流中卻默認使用它。

*來源：[Series 1](experiment-series-2.md)、[Series 2](experiment-series-2.md)*

## 發現 2：品質不是犧牲品——正確的 Prompt 下，Sonnet 超越 Opus

```
S1 策略（角色 + 深度思考指令）
Sonnet 4.6   →  15.0 / 15   成本 $0.006/任務
Opus 4.7     →  12.7 / 15   成本 $0.045/任務

品質提升 +18%  ·  成本降低 −86%
```

這不是異常值。盲測 LLM 評分（Series 2 Exp-2C）同樣發現路由模式的品質（14.1/15）與 All-Opus（14.1/15）完全相同。三個獨立方法均未發現品質損失。

**關鍵洞察**：品質的上限主要由 Prompt 策略決定，而非模型大小。對 Tier-Mid 任務，正確的 Prompt 可比 Opus 便宜 86% 且表現更好。

*來源：[Series 3](experiment-series-3.md)*

## 發現 3：路由準確率在三個獨立驗證集中均為 100%

| 驗證集 | 任務數 | 準確率 | 任務類型 |
|-------|-------|-------|---------|
| Series 2 Exp-2A | 30 | **100%** | 構造任務 × 5 技術類別 |
| Series 3 Exp-3B | 20 | **100%** | 真實 git commit 歷史 |
| Series 4 | 20 | **100%** | 4 個技術領域（前端/後端/數據/DevOps） |
| **合計** | **70** | **100%** | 構造 + 真實 + 跨域 |

**關鍵洞察**：三種不同任務來源、三次獨立測試、一致的 100% 準確率，說明 5 維度路由規則的設計本身是穩健的，不是在特定數據集上過擬合的結果。

*來源：[Series 2](experiment-series-2.md)、[Series 3](experiment-series-3.md)、[Series 4](experiment-series-4.md)*

## 發現 4：5 維度框架跨四個技術領域完全通用，評分偏差為零

AI 代理（Qwen Code）僅憑通用評分指南，對前端、後端、數據工程、DevOps 四個領域的 20 個任務評分，與人工基準的偏差：

```
J（判斷強度）  Δ = 0.00
C（上下文寬度）Δ = 0.00
R（風險權重）  Δ = 0.00
V（可驗證性）  Δ = 0.00
Cr（創意密度） Δ = 0.00
```

**關鍵洞察**：框架的 5 個維度定義足夠清晰和通用，任何具備基本工程背景的 AI 代理都可以直接應用，無需針對特定技術領域進行特殊訓練或調整。

*來源：[Series 4](experiment-series-4.md)*

## 發現 5：Haiku 能力被低估，Opus 的必要性僅在風險推理維度

```
                  Haiku 4.5  Sonnet 4.6  Opus 4.7
Tier-Exec 任務      14/15      15/15      15/15    ← Haiku 僅差 1 分
Tier-Mid 任務       14/15      13/15      13/15    ← Haiku 反超
Tier-A 任務         10/15      12/15      12/15    ← 差距 2 分，Opus 的確必要

唯一系統性差距：風險意識維度
Haiku 平均 2.0  →  Opus 平均 2.7
```

**關鍵洞察**：Opus 的必要性是窄的——只在 Tier-A 任務的「風險意識」這一個維度上有系統性優勢。對 Tier-Exec 和 Tier-Mid 任務，Haiku 可能比 Sonnet 更具性價比。這意味著路由的實際省錢空間可能比 Series 1 估計的還要大。

*來源：[Series 5](experiment-series-5.md)*

## 發現 6：R 和 J 是路由的兩個脆弱點，V 是最安全的維度

```
兩個獨立實驗一致發現：

維度          敏感率   說明
─────────────────────────────────
R（風險權重） 33%     ← 最敏感，Mid↔A 邊界最脆弱
J（判斷強度） 32%     ← 同等敏感，Series 2 精確複現
C（上下文）   16%
Cr（創意密度）13%
V（可驗證性） 11%     ← 最穩健

Series 2（30 個構造任務）J = 32%
Series 6（10 個邊界任務）J = 32%
→ 跨兩個獨立實驗的精確複現
```

**實操建議**：
- 當 R = 3 或 J = 3 時 → 觸發保守路由，直接升 Tier-A，避免 21% 的邊界路由錯誤
- V 和 Cr 的評分可以相對寬鬆，它們的誤差對路由影響最小

*來源：[Series 2](experiment-series-2.md)、[Series 6](experiment-series-6.md)*

## 綜合洞察：這 6 個發現共同說明了什麼

**gstack-plus 實驗體系回答了一個問題：「AI 任務分層路由，真的有用嗎？」**

答案是有用的，而且數據指向三個具體機制：

1. **成本節省來自任務分層，而非品質妥協**（發現 1 + 2）
2. **路由準確性由框架設計保證，而非數據偶然**（發現 3 + 4）
3. **評分困難有可預測的結構，可以提前防範**（發現 5 + 6）

這三個機制意味著 gstack-plus 不是一個需要「碰運氣」的啟發式工具，而是一個有量化保證的工程決策框架。
