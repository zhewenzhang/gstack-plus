> Phase 1 的設計總文件。整合前 8 個任務的設計決策，定義 gstack-plus 的完整架構。
>
> **狀態**：Phase 1 完成 2026-05-02

## 1. 三層模型架構

```
                    ┌─────────────────────────────────────────────┐
                    │              Tier-A（Architect）              │
                    │  模型：Claude Opus / GPT-5                   │
                    │  職責：戰略決策、架構設計、任務分解、路由     │
                    │  成本：$$$                                  │
                    └──────────────────────┬──────────────────────┘
                                           │ 任務分解 + handoff
                    ┌──────────────────────▼──────────────────────┐
                    │              Tier-Mid（Reviewer）             │
                    │  模型：Claude Sonnet / GPT-4o                │
                    │  職責：代碼審查、邏輯驗證、Exec 產出驗證     │
                    │  成本：$$                                   │
                    └──────────────────────┬──────────────────────┘
                                           │ 審查通過 / 失敗回流
                    ┌──────────────────────▼──────────────────────┐
                    │              Tier-Exec（Executor）            │
                    │  模型：Qwen Code / DeepSeek                  │
                    │  職責：代碼實現、格式重構、文檔更新          │
                    │  成本：$                                    │
                    └─────────────────────────────────────────────┘
```

**設計信念**：
- 不是所有任務都需要最強的模型。簡單任務用便宜模型，省下的成本可以用在真正需要判斷的任務上
- 關鍵是**正確路由**——把任務分配給能力恰好足夠的模型，不多也不少
- 目標：在保持質量的前提下，降低 60% 的 AI 工作流成本

## 2. Tier 職責邊界

### Tier-A（Architect）

**能做**：
- 接收用戶需求，分解為可執行任務
- 填寫 `plan-to-exec.md` 移交單（任務描述、成功標準、Scope Lock）
- 做出架構決策（方案選擇、設計方向）
- 收到 `check-to-plan.md` 回流報告後，決定後續行動
- 評估任務分類器的路由結果是否合理

**不能做**：
- 直接寫代碼（除非任務被分類為 Tier-A 執行）
- 跳過任務分類器直接路由（必須先用 5 維度評分）
- 在移交單中使用禁用詞（見 `shared/forbidden-words.md`）

### Tier-Mid（Reviewer）

**能做**：
- 審查 Exec 產出（按 `exec-to-check.md` 的 evidence 驗證）
- 發現問題時填寫 `check-to-plan.md` 回流報告
- 直接修復小問題（不改架構決策）
- 對分類器的路由結果提出異議

**不能做**：
- 修改 Tier-A 的架構決策
- 跳過 evidence 直接接受 Exec 產出
- 將回流報告的嚴重程度降級（低→中→高由 Tier-A 決定）

### Tier-Exec（Executor）

**能做**：
- 按照 `plan-to-exec.md` 的任務描述執行
- 完成後填寫 `exec-to-check.md`（必須包含 evidence）
- 遇到阻塞時報告 BLOCKED 狀態
- 建議需要超 Scope Lock 的修改（但不自行執行）

**不能做**：
- 修改 Scope Lock 以外的文件（除非在 evidence 中說明並獲得批准）
- 做出架構判斷（如「A 方案比 B 方案好」）
- 提交沒有 evidence 的完成聲明
- 使用自己的標準替換任務描述中的具體要求

## 3. 任務生命周期

```
用戶需求
  │
  ▼
┌─────────────────────────────────┐
│ Tier-A：接收需求，分解為任務      │
│ → 5 維度評分 → 路由決策           │
│ → 填寫 plan-to-exec.md           │
└──────────────┬──────────────────┘
               │ handoff
               ▼
┌─────────────────────────────────┐
│ Tier-Exec：按照移交單執行         │
│ → 實現代碼                       │
│ → 跑驗證命令                     │
│ → 填寫 exec-to-check.md          │
└──────────────┬──────────────────┘
               │ evidence + 狀態
               ▼
┌─────────────────────────────────┐
│ Tier-Mid：審查 Exec 產出          │
│ → 驗證 evidence 是否支持完成聲明  │
│ → 檢查 Scope Lock 是否遵守        │
│ → 檢查成功標準是否全部通過        │
└──────┬───────────────┬──────────┘
       │               │
   通過┘           不通過┘
       │               │
       ▼               ▼
┌──────────┐    ┌─────────────────────────────┐
│ 接受產出  │    │ Tier-Mid：填寫回流報告       │
│ → 合併    │    │ check-to-plan.md            │
│ → 通知用戶│    │ → Tier-A 決定後續行動        │
└──────────┘    └──────────────┬──────────────┘
                               │
                        重新分解/重試/升級
                               │
                               ▼
                        （回到 Tier-A 或 Tier-Mid）
```

**關鍵門控點**：
1. **路由門控**：任務必須經過 5 維度評分 + 路由規則，不能跳過
2. **證據門控**：Exec 的完成聲明必須有 evidence 支持，不能跳過
3. **審查門控**：Exec 產出必須經 Tier-Mid 審查，不能直接合併

## 4. 模板系統

```
handoff/templates/
├── plan-to-exec.md          # Tier-A → Tier-Exec 的移交單
│   ├── 任務描述（禁用詞掃描通過）
│   ├── 成功標準（可命令驗證）
│   ├── Scope Lock（明確允許/禁止的文件）
│   ├── 上下文快照
│   └── 失敗升級條件
│
├── exec-to-check.md         # Tier-Exec → Tier-Mid 的返回報告
│   ├── 完成狀態（DONE/DONE_WITH_CONCERNS/NEEDS_CONTEXT/BLOCKED）
│   ├── Evidence（commands_run + file_changes + success_criteria_check）
│   ├── Concerns（如適用）
│   └── 給 Tier-Mid 的建議
│
├── check-to-plan.md         # Tier-Mid → Tier-A 的回流報告
│   ├── 回流類型（BUILD_ERROR/LOGIC_ERROR/DESIGN_ISSUE/SCOPE_DRIFT）
│   ├── 失敗 Evidence
│   ├── 根因分析（3-strike 思維）
│   └── 建議行動
│
└── shared/
    ├── evidence-format.md   # Evidence 字段的統一規格
    └── forbidden-words.md   # 禁用詞速查卡
```

**模板關係圖**：

```
plan-to-exec.md ──→ Tier-Exec 執行 ──→ exec-to-check.md
                                                    │
                                               Tier-Mid 審查
                                                    │
                                    ┌───────────────┼───────────────┐
                                    │               │               │
                                 通過┘         小問題┘          大問題┘
                                    │               │               │
                                    ▼               ▼               ▼
                                接受產出      Tier-Mid 修復     check-to-plan.md
                                                                    │
                                                               Tier-A 回應
```

## 5. 分類器設計

### 為什麼是這 5 個維度？

| 維度 | 設計原因 |
|------|---------|
| **判斷強度** | 區分「機械操作」和「需要決策」——這是模型分工的核心維度 |
| **上下文寬度** | 區分「局部任務」和「全局任務」——context window 是有限的 |
| **風險權重** | 確保高風險任務不用便宜模型——安全/數據問題不能省錢 |
| **可驗證性** | 確保 Exec 任務有明確的成功標準——不可驗證的任務 Exec 做不好 |
| **創意密度** | 區分「套模板」和「從零設計」——創意需要更強的模型 |

這 5 個維度共同覆蓋了「任務需要什麼能力」的三個面向：
- **認知能力**：判斷強度 + 創意密度
- **知識範圍**：上下文寬度
- **風險態度**：風險權重 + 可驗證性

### 路由規則摘要

```
Tier-A:   判斷 ≥ 4 OR 風險 ≥ 4 OR 創意 ≥ 4
Tier-Exec: 判斷 ≤ 2 AND 上下文 ≤ 2 AND 可驗 ≥ 4
Tier-Mid:  以上都不是
```

詳細規則見 `classifier/routing-rules.md`。

### 保守路由原則

**不確定時，默認路由到更高 Tier。**

理由：低估任務複雜度的代價（Exec 搞砸 → 3-5 倍返工成本）遠大於高估（多用一些高階模型 token，成本多 20-50%）。

## 6. 尚未解決的設計問題

來源：`docs/learning-notes/key-insights.md` 的「開始 Phase 1 前必須解決的問題」。

### 問題 1：任務分類器的評分閾值是否合理？

**現狀**：路由規則使用固定的閾值（判斷 ≥ 4 → Tier-A），但這些閾值是基於設計推斷，不是實測數據。

**解決方案（Phase 2-3）**：通過 9 組對照實驗（3 模式 × 3 任務），收集真實的路由數據，調整閾值。

### 問題 2：失敗回流的具體升級路徑

**現狀**：`check-to-plan.md` 定義了回流格式，但「3 次失敗後升級到哪個 Tier」的決策樹還沒有完全定義。

**解決方案（Phase 2）**：在失敗回流機制的實現中定義完整的決策樹。

### 問題 3：成本追蹤和預算控制

**現狀**：沒有「今天最多花 $5」的機制。

**解決方案（Phase 2-3）**：在每次 handoff 中加入成本估算字段，在 Dashboard 中顯示累計成本。

### 問題 4：真實用戶驗證

**現狀**：所有設計都是文檔分析，沒有用戶訪談。

**解決方案（Phase 0 剩餘）**：找 5-10 個 gstack 用戶訪談，驗證假設。

## 7. Phase 1 確認節點

來源：`PROJECT_ROADMAP.md` 的 Phase 1 確認節點。

### 確認節點 1：路由表能對 20 個測試任務做出合理分配

**驗證方法**：
1. 拿出 `classifier/test-tasks.md` 中的每個任務
2. 只看 `classifier/scoring-guide.md` 評分
3. 只看 `classifier/routing-rules.md` 路由
4. 比較結果和任務中標註的「正確路由」
5. 目標：≥ 80% 一致率（16/20）

**當前狀態**：✅ 測試集已創建，路由規則已定義。待用戶驗證。

### 確認節點 2：模板系統能生成高質量的 handoff

**驗證方法**：
1. 用 `plan-to-exec.md` 模板為一個真實任務生成移交單
2. 檢查：任務描述是否不含禁用詞？成功標準是否可驗證？Scope Lock 是否明確？
3. 用 `exec-to-check.md` 模板模擬 Exec 返回
4. 用 `check-to-plan.md` 模板模擬回流

**當前狀態**：✅ 模板已創建，格式已定義。待實際使用驗證。

### 確認節點 3：分類器和模板系統集成

**驗證方法**：
1. 給一個任務描述
2. 用分類器評分 + 路由 → 得到目標 Tier
3. 用對應模板生成 handoff
4. 檢查全流程是否通暢

**當前狀態**：⏳ 需要 Phase 2 的代碼實現來驗證集成。
