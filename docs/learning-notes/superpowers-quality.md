## 一、`verification-before-completion` — 完成前驗證

### 字面層：它做了什麼？

這個技能定義了一個鐵律：**在任何「完成、修復、通過」的聲稱之前，必須運行驗證命令並確認輸出。**

**Gate 函數（5 步）：**

```
BEFORE 聲稱任何狀態或表達滿意：

1. IDENTIFY: 什麼命令能證明這個聲稱？
2. RUN: 執行完整命令（新鮮的、完整的）
3. READ: 完整輸出，檢查退出碼，計數失敗
4. VERIFY: 輸出確認了聲稱嗎？
   - 否 → 陳述實際狀態 + 證據
   - 是 → 陳述聲稱 + 證據
5. 只有到這一步：才能做出聲稱

跳過任何一步 = 撒謊，不是驗證
```

**常見失敗模式**：

| 聲稱 | 需要 | 不足夠的 |
|------|------|---------|
| 測試通過 | 測試命令輸出：0 失敗 | 之前的運行、「應該通過」 |
| Linter 乾淨 | Linter 輸出：0 錯誤 | 部分檢查、推斷 |
| 構建成功 | 構建命令：exit 0 | Linter 通過、日誌看起來不錯 |
| Bug 修復了 | 測試原始症狀：通過 | 代碼改了、假設修好了 |
| Agent 完成了 | VCS diff 顯示改動 | Agent 報告「成功」 |

**紅旗（立即停止）**：
- 使用 "should"、"probably"、"seems to"
- 在驗證之前表達滿意（"Great!"、"Perfect!"、"Done!"）
- 準備 commit/push/PR 但沒有驗證
- 信任 agent 的成功報告
- 依賴部分驗證
- 想著「就這一次」
- 累了想結束工作
- **任何暗示成功但沒有運行驗證的措辭**

**理性化預防表**：

| 藉口 | 現實 |
|------|------|
| "應該能行了" | 運行驗證 |
| "我有信心" | 信心 ≠ 證據 |
| "就這一次" | 沒有例外 |
| "Linter 過了" | Linter ≠ 編譯器 |
| "Agent 說成功了" | 獨立驗證 |
| "我累了" | 疲勞不是藉口 |
| "部分檢查夠了" | 部分證明不了任何事 |
| "換個說法規則就不適用" | 精神重於字面 |

### 設計層：為什麼這樣設計？

#### 設計決策：「證據先於斷言」——這比 gstack 的 `/qa` 有什麼不同？

**核心差異**：gstack 的 `/qa` 是「QA 工程師角色」——它用瀏覽器像真實用戶一樣測試，找到 bug 後自動修復。superpowers 的 `verification-before-completion` 不是角色，而是**紀律規則**。

gstack 的 `/qa` 說：「你是 QA 工程師，用瀏覽器測試，找到 bug，修復，原子 commit。」
superpowers 的 `verification-before-completion` 說：「在你聲稱完成之前，運行命令，讀輸出，然後才能說完成。」

**gstack 是流程，superpowers 是紀律**。

gstack 的 `/qa` 依賴角色定義來保證質量——當 AI 被告訴「你是 QA 工程師」，它會按照 QA 的方式工作。superpowers 不依賴角色——它依賴硬性規則：「沒有運行驗證命令就不能說完成」。

**為什麼 superpowers 選擇紀律而不是角色**：

因為 superpowers 觀察到了 24 次失敗記憶，其中最痛苦的失敗模式是：AI 聲稱完成了但實際沒有——測試沒跑過、函數沒定義、需求沒實現。這不是「角色不夠好」的問題，是「AI 在沒有證據的情況下做出了聲稱」。

**反例**：AI 改了代碼，沒有跑測試，說「測試應該通過了」。結果：undefined function 被推送到生產，應用崩潰。如果有 verification-before-completion，AI 會跑測試，看到失敗，修復，然後才說完成。

## 二、`systematic-debugging` — 系統性調試

### 字面層：它做了什麼？

`systematic-debugging` 是四階段的根因調試流程。鐵律：**沒有根因調查就不能修復**。

**四個階段（必須按順序完成）：**

**Phase 1: Root Cause Investigation**
```
1. 仔細讀錯誤信息（不要跳過警告）
2. 一致性地重現（能穩定觸發嗎？精確步驟？）
3. 檢查最近改動（git diff、新依賴、配置變化）
4. 多組件系統中收集證據
   → 在每個組件邊界日誌輸入/輸出
   → 運行一次，找到在哪裡斷掉
5. 追蹤數據流（從錯誤位置向上追溯到源頭）
```

**Phase 2: Pattern Analysis**
```
1. 找到代碼庫中類似的正常工作代碼
2. 完整讀參考實現（不要跳讀）
3. 列出所有差異（無論多小）
4. 理解依賴關係（需要什麼組件、配置、環境？）
```

**Phase 3: Hypothesis and Testing**
```
1. 形成單一假設（「我認為 X 是根因因為 Y」）
2. 最小化測試（只改一個變量）
3. 驗證後再繼續
   → 成功 → Phase 4
   → 失敗 → 形成新假設（不要在頂上加更多修復）
4. 不知道就說不知道（問、研究）
```

**Phase 4: Implementation**
```
1. 創建失敗的測試用例（最簡單的重現）
2. 實施單一修復（只解決根因，沒有「既然在這裡順便改改」）
3. 驗證修復（測試過了嗎？其他測試沒壞嗎？問題真的解決了嗎？）
4. 如果修復不工作：
   → < 3 次 → 回到 Phase 1
   → ≥ 3 次 → 停止 → 質疑架構（第 5 步）
5. 3+ 次失敗 → 質疑架構
   → 每個修復都揭示新問題 → 架構錯了
   → 和用戶討論是否重構架構
```

**Red Flags（立即停止並回到 Phase 1）**：
- "Quick fix for now, investigate later"
- "Just try changing X and see"
- "Add multiple changes, run tests"
- "Skip the test, manually verify"
- "Probably X, fix that"
- "One more fix attempt"（已經試過 2+ 次時）
- 每個修復都在不同地方揭示新問題

### 設計層：為什麼這樣設計？

#### 設計決策 1：與 gstack 的 `/investigate` 對比，兩者有什麼核心差異？

**相似之處**：
- 都要求先找根因再修復（Iron Law 完全一致）
- 都有 3-strike 規則（3 次失敗 → 停止 → 質疑架構）
- 都要求最小化修復（不要 "while I'm here" 改動）
- 都要求回歸測試（沒修復時 FAIL，有修復時 PASS）

**核心差異**：

| 維度 | gstack `/investigate` | superpowers `systematic-debugging` |
|------|----------------------|-----------------------------------|
| Scope Lock | PreToolUse hook 物理限制編輯範圍 | 沒有物理限制，靠紀律 |
| Pattern 分類 | 6 種已知模式（Race condition、Nil propagation 等） | 找代碼庫中類似工作代碼 |
| 多組件調試 | 沒有專門處理 | Phase 1.4：在每個組件邊界日誌輸入輸出 |
| 外部搜索 | `gstack-learnings-search` 跨項目搜索 | 沒有跨項目搜索 |
| 報告格式 | 標準化 DEBUG REPORT 模板 | 沒有標準報告格式 |

**superpowers 的獨特設計**：Phase 1.4 的「多組件證據收集」——在每個組件邊界加日誌，運行一次，找到在哪裡斷掉。這個設計比 gstack 的 `/investigate` 更系統性——gstack 假設問題在代碼裡，superpowers 假設問題可能在「數據流經過的任何地方」（環境、配置、組件間通信）。

**gstack 的獨特設計**：Scope Lock 通過 PreToolUse hook 強制執行。superpowers 沒有這種物理強制——它靠紀律和 Red Flags 表來防止 scope creep。

**為什麼差異**：gstack 設計給 Claude Code（有 hooks 能力），superpowers 設計給多平台（Claude Code、Copilot CLI、Gemini CLI）。PreToolUse hooks 是 Claude Code 特有的，不能跨平台使用。所以 superpowers 選擇了「紀律 + Red Flags」而不是「物理 hook」。

---

#### 設計決策 2：「3+ 次失敗 → 質疑架構」——這不是假設錯了，是架構錯了

**決策**：當 3 個假設都失敗時，不是繼續形成新假設，而是停下來質疑整個模式/架構。

**為什麼**：如果每次修復都揭示新問題（修了 A 發現 B 壞了，修了 B 發現 C 壞了），這不是代碼錯了——是設計錯了。繼續修復只是在症狀上貼創可貼。

**識別模式**：
- 每個修復都揭示新的共享狀態/耦合/不同地方的問題
- 修復需要「大規模重構」才能實現
- 每個修復都在別處創造新症狀

這些是在說：「這個架構 fundamentally 有問題」，不是「這段代碼有 bug」。

**反例**：3 次修復都失敗了，繼續第 4 次。第 4 次也失敗了。浪費了 4 倍時間，最後發現是整個數據流設計錯了——應該從頭重構，不是在錯誤的基礎上修補。

## 三、`test-driven-development` — 測試驅動開發

### 字面層：它做了什麼？

superpowers 的 TDD 是經典的 Red-Green-Refactor 循環，但規則極其嚴格。

**鐵律**：

```
沒有失敗測試就不能寫生產代碼
```

如果先寫了代碼再寫測試 → **刪除代碼，從零重來**。

**Red-Green-Refactor 循環**：

```
RED → 寫失敗的測試
  ↓
Verify RED → 確認失敗（不是報錯，是預期失敗）
  ↓
GREEN → 寫最小代碼讓測試通過
  ↓
Verify GREEN → 確認通過（且其他測試沒壞）
  ↓
REFACTOR → 清理代碼（不添加行為）
  ↓
Next → 下一個失敗測試
```

**關鍵規則**：

1. **刪除先寫的代碼**：如果先寫了實現代碼，不是「補測試」，是「刪除實現，從測試重來」。
2. **沒有例外**：不能作為「參考」保留、不能「邊看邊改編」、不能「先探索再測試」。
3. **Verify RED 是強制**：測試必須因為「缺少功能」而失敗，不是因為拼寫錯誤或語法錯誤。
4. **測試後置 ≠ TDD**：測試後置回答「這段代碼做什麼」；測試前置回答「這段代碼應該做什麼」。

**為什麼順序重要（5 個常見藉口及其反駁）**：

| 藉口 | 反駁 |
|------|------|
| "我寫完再測試來驗證" | 測試立刻通過證明不了什麼——可能測試了錯誤的東西 |
| "我已經手動測試了所有邊界情況" | 手動測試是即興的，沒有記錄，不能重跑 |
| "刪除 X 小時的工作是浪費" | 沉沒成本謬誤。保留沒有測試的代碼才是技術債 |
| "TDD 是教條主義，務實意味著適應" | TDD 就是務實：發現 bug 更快、防止回歸、記錄行為 |
| "測試後置能達到同樣目的" | 不。測試後置驗證「你記住了所有邊界情況」。測試前置發現「你忘記的邊界情況」 |

### 設計層：為什麼這樣設計？

#### 設計決策 1：superpowers 的 TDD 和傳統 TDD 有什麼不同？

**傳統 TDD**：是一種開發方法，建議先寫測試。但很多團隊會「靈活處理」——有時候先寫代碼再補測試，有時候跳過測試。

**superpowers 的 TDD**：是**紀律規則**，不是方法論。它說：
- 先寫了代碼？刪除，重來。
- 想著「就這一次」？那是理性化，停止。
- 覺得「太簡單了不需要測試」？簡單代碼也會壞，測試只要 30 秒。

**核心差異**：傳統 TDD 是「最佳實踐」，superpowers 的 TDD 是「不可違背的規則」。

**為什麼**：因為 superpowers 從失敗記憶中學到：當 AI「靈活處理」TDD 時，它幾乎總是選擇了「先寫代碼再補測試」——而補的測試幾乎總是「立刻通過」的無效測試（測試了已有的行為而不是預期的行為）。

**反例**：AI 先寫了 `retryOperation` 函數，然後寫測試。測試立刻通過，因為測試寫的是「調用 3 次 mock」而不是「真實的重試行為」。測試覆蓋率 100%，但測試沒有發現任何問題。

---

#### 設計決策 2：這三個技能共同反映了什麼信念？

**核心信念：AI 傾向於聲稱完成但實際沒有驗證（自我欺騙）。**

這三個技能都在防止同一種失敗模式：

1. `verification-before-completion`：防止「聲稱完成但沒有跑驗證」
2. `systematic-debugging`：防止「聲稱修復但沒有找到根因」
3. `test-driven-development`：防止「聲稱有測試但測試是事後補的」

它們共同反映了一個痛苦的觀察：**AI 會「覺得」自己完成了，因為它改了代碼，但它沒有實際運行代碼看結果。**

**為什麼 AI 會這樣**：LLM 的訓練方式讓它傾向於「基於模式匹配預測下一步」而不是「執行並觀察結果」。當它寫了「正確的代碼」（根據訓練數據的模式），它「相信」這是正確的——但它沒有實際執行，所以不知道在當前環境中是否真的正確。

**這三個技能的共同設計模式**：
- **鐵律**：每個技能都有不可違背的規則（NO COMPLETION CLAIMS, NO FIXES WITHOUT ROOT CAUSE, NO PRODUCTION CODE WITHOUT FAILING TEST）
- **Red Flags**：列出 AI 試圖跳過紀律時的「理性化想法」
- **強制順序**：不能跳過步驟，不能「先做 X 稍後做 Y」
- **獨立驗證**：不信任 AI 自己的判斷，必須運行命令看輸出

## 哲學層：反映了什麼信念？

**信念 1：AI 的「自信」和「正確」是兩回事。**

三個技能都在說同一件事：AI 覺得對了不算數。測試跑了並通過才算數。根因找到了才算數。代碼在當前環境裡跑了沒問題才算數。

這不是不信任 AI——是信任證據。AI 的訓練數據讓它善於預測「看起來正確的代碼」，但預測不等同於驗證。

**信念 2：紀律比靈活性更重要。**

這三個技能都沒有「根據情況調整」的選項。它們是強制性的、不可跳過的、沒有例外的。因為超能力的「超」不在於「更靈活」，而在於「更紀律」。

**信念 3：「簡單」和「緊急」不是跳過紀律的理由。**

「這個問題很簡單」→ 簡單問題也有根因。
「緊急修復」→ 系統性調試比猜測更快。
「就這一次」→ 沒有例外。

紀律的價值恰恰在最想跳過它的時刻體現——因為那些時刻跳過它的代價最大。

---

### 三個技能缺一不可：刪掉任何一個的後果

| 保留哪兩個 | 刪掉哪個 | 具體後果 |
|-----------|---------|---------|
| verification + debugging | TDD | AI 寫的代碼「能跑」，但沒有預先定義失敗標準。AI 可以寫讓測試通過的任何實現，包括針對已知輸入的 hardcode 返回值。測試通過不等於邏輯正確。 |
| TDD + debugging | verification | AI 按 TDD 流程走，但宣告完成前沒有跑完整測試套件。「這個新測試通過了」不代表「之前的 32 個測試還在通過」。迴歸 bug 在 review 時才被發現。 |
| TDD + verification | debugging | 測試失敗時，AI 反覆猜測修改方向（先改 A，不行再改 B），而不是系統地隔離根因。3 次嘗試後問題可能更複雜，因為修改疊加了新的副作用。 |

---

## 對比 gstack：superpowers 的質量技能更輕量，gstack 的 `/qa` 很重。這個取捨說明了什麼？

**gstack 的 `/qa` 是「重量級流程」**：
- 瀏覽器自動化（持久 Chromium）
- 測試受影響的每個頁面
- Health Score（8 個類別加權）
- Fix Loop（每個 bug 原子 commit）
- Regression test 生成

**superpowers 的質量技能是「輕量級紀律」**：
- 沒有瀏覽器——只是運行命令看輸出
- 沒有 health score——只是測試通過/失敗
- 沒有自動化修復——只是找到根因手動修復
- 沒有覆蓋率審計——只是 TDD 循環

**這個取捨說明了兩個系統的不同定位**：

| | gstack | superpowers |
|--|--------|-------------|
| 質量方法 | 流程（QA 工程師角色） | 紀律（規則 + 驗證） |
| 測試深度 | 端到端（瀏覽器 + 用戶流程） | 單元（TDD + 命令驗證） |
| 自動化程度 | 高（自動修復、原子 commit） | 低（人工修復、紀律保證） |
| 適用場景 | Web 應用、UI 密集 | 任何代碼、庫、CLI |
| 成本 | 高（瀏覽器、多個 specialist） | 低（只是跑命令） |

**gstack 的 `/qa` 適合**：有 UI 的 web 應用，需要確保用戶流程正常工作。
**superpowers 的質量技能適合**：任何代碼庫，特別是後端、庫、CLI——沒有 UI 可以瀏覽器測試的場景。

**更深層的洞察**：gstack 假設用戶在做 web 應用（所以有瀏覽器基礎設施）；superpowers 假設用戶在做「任何編程任務」（所以只有通用的命令驗證）。gstack 更垂直，superpowers 更水平。

## 對 gstack-plus 的啟發

### 啟發 1：「證據先於斷言」是 Exec 模型輸出的核心驗證規則

gstack-plus 的 Exec 模型（Qwen）完成任務後返回「已完成」。在聲稱被接受之前，必須獨立驗證——不能只相信 Exec 的報告。

**具體應用**：在 handoff 模板中加入驗證步驟：
```
Exec 返回結果
  → Claude 獨立驗證:
       ├── 跑測試（不是 Exec 跑的，是 Claude 跑的）
       ├── 檢查 diff（改動符合任務描述嗎？）
       ├── 檢查範圍（沒有超出允許改動的文件？）
       └── 全部通過 → 接受
           → 任何失敗 → 退回 Exec 重新修復
```

### 啟發 2：系統性調試的「多組件證據收集」可以用於跨模型協作的失敗診斷

當 Exec 模型的輸出有問題，不是直接「修復」——而是在 Claude 和 Exec 之間加診斷日誌：
- Claude 發送的指令是什麼？
- Exec 接收到的指令是什麼？
- Exec 的輸出是什麼？
- Claude 接收到的輸出是什麼？

這能幫助找到問題在哪個環節（指令不清？Exec 誤解？執行錯誤？）。

### 啟發 3：TDD 的「刪除先寫的代碼」規則太嚴格，但「測試前置」的思想值得借鑒

**TDD 精神在 gstack-plus 中的具體含義——Spec-first 執行**：

不要求 Qwen 先寫測試（Qwen 可能無法訪問測試框架），但要求：

**步驟 1**：Claude 在派發任務時，在 prompt 中加入「成功標準」字段：
```
成功標準：
- 當輸入為 null 時，函數返回 []
- 當輸入為空數組 [] 時，函數返回 []
- 當輸入為 [1, 2, 3] 時，函數返回過濾後的結果
- 運行 npm test src/utils/ 全部通過
```

**步驟 2**：Qwen 完成後，Claude 逐條驗證成功標準，不接受「邏輯上應該滿足」的聲明。

**為什麼這比「TDD 精神」更有用**：
它把「測試優先的思維」從 Qwen（不適合做設計決策）移到 Claude（應該定義成功標準）。
每個標準都可以用命令驗證（Falsifiable），排除了「代碼看起來正確」的主觀判斷。

### 啟發 4：3-strike 規則的通用性

三個質量技能都有「3 次失敗就停止」的模式。這是超能力的通用模式：
- 驗證 3 次不過 → 停止，重新分析
- 調試 3 個假設失敗 → 停止，質疑架構
- TDD 不是重點，但 3 次重寫還是不過 → 停止，重新設計

gstack-plus 的失敗回流也應該用這個模式：Exec 失敗 3 次 → 升級到 Tier-Mid 分析 → 再失敗 → 升級到 Tier-A。

### 啟發 5：「紀律 > 角色」的取捨

gstack 用角色（「你是 QA 工程師」）保證質量，superpowers 用紀律（「沒有驗證不能說完成」）保證質量。gstack-plus 可以結合兩者：
- 給每個 Tier 定義角色（Architect、Reviewer、Executor）
- 但用紀律規則保證每個角色的輸出質量（不是依賴角色自律）
