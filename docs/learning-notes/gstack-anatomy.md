# gstack 解剖學：設計哲學綜合分析

> Day 7 | 綜合分析 gstack-overview、gstack-planning-skills、gstack-review-skills、gstack-shipping-skills、ETHOS.md

---

## TL;DR：gstack 設計哲學（5 句話）

> **gstack 把 AI 輔助開發從「讓 AI 寫代碼」重新定義為「讓 AI 在正確的時刻做出正確的判斷」。**
> 
> 它通過 26+ 個「角色化技能」實現——每個技能不是工具，而是一個特定的專家視角（CEO、Engineer Manager、QA Lead、Security Officer），在正確的時機被激活。
> 
> 所有技能共享三個核心原則：「做完整的事」（Boil the Lake，因為 AI 讓完整性幾乎免費）、「建之前先搜索」（避免重複造輪子）、「AI 建議，用戶決定」（User Sovereignty，人類始終保持最終判斷權）。
> 
> 它優化的是「決策的質量和一致性」，而不是「代碼產能」——因為在 AI 時代，寫代碼已經太便宜了，但知道寫什麼仍然很貴。
> 
> 代價是速度和靈活性——gstack 的每次工作流都很徹底，不適合需要快速原型的場景。但 gstack 的信念是：在 AI 時代，「快速原型」和「完整實現」的成本差異已經縮小到了值得每次都做完整的事。

## 一、回答 LEARNING_PLAN.md 的 7 個核心問題

### Q1：gstack 為什麼這樣設計？（不是「怎麼做」，是「為什麼」）

**答案**：gstack 的設計來自一個核心信念——**AI 時代的瓶頸從「代碼產能」轉移到了「判斷質量」**。

ETHOS.md 的開篇就說明了背景：「一個人用 AI 現在可以完成過去需要 20 人團隊的工作量。工程屏障已經消失。剩下的是品味、判斷力和把整件事做完的意願。」

這解釋了所有設計決策的「為什麼」：

- **為什麼用角色而不是工具？** 因為當代碼產能不再是瓶頸，最大的風險是「快速地做了錯誤的事」。角色化（「你是偏執的 Staff Engineer」）強制 AI 用特定視角思考，而不只是「寫代碼」。
- **為什麼每個技能都有冗長的 preamble？** 因為 preamble 不是「初始化代碼」——它是「讓 AI 進入正確心態」的強制步驟。 telemetry、session tracking、feature discovery 都在確保 AI 知道「我正在哪個 session 裡工作，我有什麼上下文」。
- **為什麼有 26+ 個技能而不是 3 個大技能？** 因為每個技能對應一個「判斷時刻」。什麼時候需要 CEO 視角（方向對了嗎）？什麼時候需要 Eng Manager 視角（架構對嗎）？什麼時候需要 QA 視角（功能對嗎）？它們是不同的問題，不能用同一個技能回答。

**gstack 不是「幫助 AI 寫代碼」——它是「幫助 AI 在正確的時間做正確的判断」。**

### Q3：Brain Sync 解決了什麼根本問題？

從 gstack-overview 的筆記中，GBrain Sync 是基於 GitHub repo 的跨機器記憶同步機制。

**它解決的根本問題是**：AI 的工作記憶是 session 級的（每次會話結束後清零），但項目的「操作知識」是持久的（「這個 repo 的測試要用 `bun test`，不是 `npm test`」）。如果不外化存儲，AI 每次 session 都從零開始，重複犯同樣的「新手錯誤」。

**更深層的問題**：當同一個開發者在不同機器上工作（家裡的 MacBook、辦公室的 PC），AI 在不同機器上學到的項目知識不會自動共享。GBrain Sync 通過 GitHub 作為中介，讓 `learnings.jsonl` 跨機器同步。

**這反映了一個更大的設計理念**：AI 的知識應該外化為「可讀取的持久化資產」，而不是依賴於模型訓練或 prompt。learnings 系統是 gstack 的「長期記憶」。

### Q4：「Boil the Lake」原則的真實含義？

**字面含義**：AI 讓「做完整的事」的邊際成本幾乎為零。當完整實現比捷徑只多幾分鐘，就做完整的。

**真實含義**：這是一個 **「完整性 vs 捷徑」的經濟學重估**。

在 AI 之前，完整實現（100% 測試覆蓋、所有邊緣情況處理、完整文檔更新）需要 2 天，捷徑（90% 覆蓋、主要路徑處理、文檔以後再補）需要 4 小時。選擇捷徑是經濟理性的——節省了 1.75 天。

AI 讓完整實現需要 15 分鐘，捷徑需要 8 分鐘。差異是 7 分鐘。選擇捷徑不再經濟理性——你省了 7 分鐘，但欠下了永久的技術債。

**ETHOS.md 的壓縮比表格說明了為什麼**：

| 任務類型          | 人類團隊 | AI 輔助 | 壓縮比   |
| ------------- | ---- | ----- | ----- |
| Boilerplate   | 2 天  | 15 分鐘 | ~100x |
| 測試編寫          | 1 天  | 15 分鐘 | ~50x  |
| Bug 修復 + 回歸測試 | 4 小時 | 15 分鐘 | ~20x  |
| 架構/設計         | 2 天  | 4 小時  | ~5x   |
| 研究/探索         | 1 天  | 3 小時  | ~3x   |

**關鍵洞察**：壓縮比不是均勻的。越是「機械化」的工作，AI 的加速越大（100x）；越是「需要判斷」的工作，加速越小（3-5x）。這意味著：**在 AI 時代，「判斷」的相對價值上升了，「產能」的相對價值下降了**。

**Lake vs Ocean 的區分也很關鍵**：

- Lake（可燒乾的）：一個模塊的 100% 測試覆蓋、一個功能的完整實現、所有錯誤路徑
- Ocean（不可燒乾的）：從零重寫整個系統、跨季度的平台遷移

**真實含義的最後一層**：Boil the Lake 不是「做所有事」——它是「在可燒乾的範圍內做完整的事」。識別什麼是 Lake、什麼是 Ocean，本身就是一種判斷力。

### Q5：為什麼 skill 是 markdown 而不是代碼？

**答案來自三個層面**：

**1. Markdown 是「工作流」的自然語言**

技能描述的是「步驟 → 判斷 → 分支 → 輸出」的工作流，不是「輸入 → 計算 → 輸出」的算法。工作流用自然語言表達最清晰，用代碼表達會變得笨重。

例如，`/review` 的 Scope Drift Detection：

- 用 Markdown：「如果 diff 包含了計劃中沒有的功能 → 標記為 SCOPE CREEP」
- 用代碼：需要解析 plan 文件、解析 diff、做文本匹配、處理邊緣情況、處理 plan 文件不存在的情況...

**2. Markdown 可讀、可審計、可修改**

任何用戶都可以打開 SKILL.md 讀懂「這個技能在做什麼」。如果是代碼，用戶需要理解腳本邏輯。如果是編譯後的二進制，用戶完全不知道技能在做什麼。這與 User Sovereignty 原則一致——用戶應該能審計 AI 的行為。

**3. Markdown 是 LLM 的原生格式**

LLM 讀寫 Markdown 的能力遠強於讀寫代碼。用 Markdown 定義技能意味著：

- 用戶可以直接修改技能行為（改文字，不需要改代碼邏輯）
- 技能可以通過 prompt 注入到 AI 的上下文中
- 技能的更新可以是 git diff，人類和 AI 都能理解

**反例**：如果 skill 是 Python 腳本，用戶需要懂 Python 才能修改技能行為；如果是 YAML 配置，表達力不足以描述複雜工作流；如果是 prompt template，缺乏條件邏輯和循環結構。

### Q6：gstack 和傳統的 prompt engineering 區別在哪？

**傳統 prompt engineering**：

```
「你是一個有經驗的軟件工程師。請 review 這段代碼，找出潛在的問題。」
```

**gstack 的 `/review` 技能**：

```
1. 檢測平台和 base branch
2. 確認在 feature branch 上（否則停止）
3. Scope Drift Detection（計劃 vs diff 對比）
4. 讀 checklist.md（不可跳過）
5. 獲取 diff
6. Critical Pass + 置信度校準
7. Review Army（並行 specialist dispatch）
8. 合併 findings，PR Quality Score
9. Fix-First 分類
```

**核心區別**：

| 維度   | Prompt Engineering           | gstack                                   |
| ---- | ---------------------------- | ---------------------------------------- |
| 結構   | 一段文字                         | 結構化的工作流（步驟 + 判斷 + 輸出）                    |
| 可驗證性 | 「AI 做了什麼？」不清楚                | 每個步驟有具體的 bash 命令和輸出                      |
| 一致性  | 每次 AI 可能做不同的事                | 相同的步驟順序，相同的輸出格式                          |
| 可審計性 | 難以檢查 AI 是否真的執行了 prompt 的所有部分 | 每個步驟的輸出可以審核                              |
| 角色深度 | 「你是工程師」→ AI 自行發揮             | 「你是偏執的 Staff Engineer，用這 15 個認知模式」→ 具體框架 |
| 失敗處理 | 沒有「如果步驟 X 失敗怎麼辦」             | 每個步驟有錯誤處理和 fallback                      |
| 知識積累 | 每次從零開始                       | learnings 系統跨 session 共享                 |

**最大的區別**：prompt engineering 是「告訴 AI 你想要什麼」；gstack 是「定義 AI 應該如何工作」。前者是一次性指令，后者是結構化紀律。

### Q7：哪些設計決策是「對的」？哪些可以改進？

**對的決策**：

1. **角色化 > 工具化**：讓 AI 成為「偏執的 Staff Engineer」比「review 代碼」效果好得多。這是 LLM 的本質特徵——它們對「角色指令」的響應遠好於「功能指令」。

2. **Review Readiness Dashboard**：可視化哪些審查跑了、哪些沒跑、哪些過時了。這讓「流程完整性」變成可見的，而不是「我大概跑了吧」。

3. **Fix-First 哲學**：顯而易見的問題自動修復，模糊的才問人。這減少了認知負擔——用戶不需要判斷每個問題是「簡單的」還是「需要我決定的」。

4. **Learnings 系統（JSONL）**：項目級知識的外化存儲。沒有這個，AI 每次 session 都從零開始。

5. **User Sovereignty 的具體實現**：所有 gate 都是「問你」不是「阻止你」，AskUserQuestion 的固定格式確保用戶得到充分信息。

**可以改進的**：

1. **技能的 DRY 問題**：26+ 個技能之間有大量重複（preamble、平台檢測、learnings 搜索）。雖然有模板系統，但工作流邏輯本身也在多個技能中重複（如「檢測 runtime → 選擇框架 → 安裝」在 `/ship` 和 `/qa` 中都出現）。**改進**：提取「子技能」機制，讓技能可以調用其他技能的核心邏輯。

2. **模型單一性**：整個 gstack 只用 Claude。雖然有 `/codex` 作為第二意見，但核心工作流是 Claude 獨角戲。**改進**：gstack-plus 的多模型協作就是對這個問題的回應。

3. **沒有成本追蹤**：雖然有 telemetry，但沒有「這個 session 花了多少錢」的視圖。**改進**：在每個技能結束時輸出成本估算（token 使用量 × 模型單價）。

4. **任務路由是手動的**：用戶需要知道 `/investigate` 適合 bug、`/review` 適合代碼審查。**改進**：gstack-plus 的任務分類器應該自動識別任務類型並路由。

## 二、gstack 假設了什麼樣的用戶？

從所有技能的設計中可以提取出 gstack 的「用戶畫像」：

**核心用戶**：**「AI 增強的獨立開發者」或「小團隊的技術負責人」**。

證據：

- **Solo vs Collaborative 模式**：`/ship` 的 Test Failure Triage 區分了 solo 和 collaborative，推薦完全不同。Solo 開發者是主要場景。
- **「一個人完成 20 人團隊的工作」**：ETHOS.md 的核心敘事。
- **不需要編程基礎設施知識**：`/ship` 會自動 bootstrap 測試框架、CI/CD、文檔。用戶不需要知道「怎麼建 jest.config.js」——AI 知道。
- **但需要產品判斷力**：`/office-hours` 要求用戶回答 6 個逼問問題，假設用戶有產品洞察。
- **有 git 但不一定是 git 專家**：技能自動處理 merge、rebase、cherry-pick，但用戶需要理解「為什麼要這樣」。

**用戶的主要風險**：

1. **速度太快，方向錯了**：AI 讓代碼產能變大，但方向錯誤時代價也變大。→ CEO review、office-hours
2. **完整性不足**：AI 生成的代碼看起來對，但缺少測試、文檔、邊緣情況。→ Boil the Lake、QA 流程
3. **過度信任 AI 的輸出**：用戶可能認為 AI 生成的代碼就是正確的。→ Anti-sycophancy、Review Army、Fix-First
4. **知識不積累**：每次 session 都從零開始。→ Learnings 系統

## 三、gstack 優化什麼？犧牲什麼？

**優化**：

1. **決策質量 > 代碼產能**：26+ 個審查封印 vs 3 個核心執行技能
2. **完整性 > 速度**：Boil the Lake、100% 測試覆蓋率目標
3. **一致性 > 靈活性**：固定步驟順序、固定輸出格式
4. **可審計性 > 自動化**：AskUserQuestion 在關鍵節點，而不是全自動

**犧牲**：

1. **速度**：26+ 個技能的學習和執行需要時間。`/autoplan` 跑完 CEO → Design → Eng → DX 可能需要 30+ 分鐘。
2. **靈活性**：固定的步驟順序不適合所有場景。有時候你只需要「快速看一下代碼」，不需要完整的 Review Army。
3. **認知負擔**：用戶需要理解 26+ 個技能的存在、用途和調用時機。學習曲線陡峭。
4. **成本**：26+ 個技能 + Review Army + Learnings 搜索 = 大量 token 消耗。

**核心取捨**：gstack 選擇了「每次都很徹底」而不是「有時快速有時徹底」。這是合理的——因為 AI 讓「徹底」的成本大幅下降——但它意味著不適合所有場景。

## 六、從解剖學看 gstack-plus 的設計空間

基於以上 7 個問題的回答，gstack-plus 的核心增量價值可以定位為：

**gstack 解決了「人類 + AI 如何做出正確決策」的問題，但假設了「決策後只用一個模型執行」。**

**gstack-plus 要解決的是「決策後如何用多個模型協作執行，在保持質量的前提下降低成本」**。

具體來說：

1. **gstack 的角色化** → gstack-plus 可以繼承：每個 Tier 都應該是明確的角色
2. **gstack 的 Boil the Lake** → gstack-plus 可以繼承：Exec 模型也應該做完整的事
3. **gstack 的 User Sovereignty** → gstack-plus 需要擴展：用戶不僅決定「方向」，還能看到「哪個模型在做什麼」
4. **gstack 的 Learnings 系統** → gstack-plus 需要擴展：跨模型的學習（「Exec 模型在這類任務上容易犯什麼錯誤」）
5. **gstack 的單一模型** → gstack-plus 的核心創新：模型分工

**gstack-plus 不應該改變的**：

- 角色化的設計理念（但角色定義需要調整以適應多模型）
- 結構化的工作流程（步驟 + 判斷 + 輸出）
- Fix-First 哲學
- User Sovereignty

**gstack-plus 需要新增的**：

- 任務分類器（自動識別任務類型 → 路由到正確模型）
- 失敗回流機制（Exec 失敗 → 智能升級，不是從零重來）
- 成本追蹤（每個模型的 token 消耗和成本）
- 跨模型 Learnings（「這類任務 Exec 模型的常見錯誤模式」）
