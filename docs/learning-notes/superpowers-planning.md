## 一、`using-superpowers` — 強制 Invoke 規則

### 字面層：它做了什麼？

`using-superpowers` 不是工作流技能，而是**元規則**——定義了所有 superpowers 技能的調用紀律。

**核心規則（1% 規則）**：

```
「如果你認為有 1% 的機率某個技能適用於你正在做的事，你絕對必須 invoke 該技能。」
```

這不是可選的，不是建議，是**強制**。

**紅旗表（AI 自我合理化的模式）：**

| AI 的想法 | 現實 |
|---------|------|
| "這只是個簡單問題" | 問題也是任務，檢查技能 |
| "我需要先了解更多" | 技能檢查在澄清問題之前 |
| "我可以先看看代碼" | 技能告訴你怎麼看 |
| "這個不需要正式技能" | 如果有技能，就用它 |
| "我記得這個技能怎麼用" | 技能會演化，讀當前版本 |
| "這個技能殺雞用牛刀" | 簡單的事會變複雜，用它 |
| "我先做這一件小事" | 在做任何事之前檢查技能 |

**技能優先級**：
1. **用戶明確指令**（CLAUDE.md, AGENTS.md）— 最高
2. **Superpowers 技能** — 覆蓋系統默認行為
3. **系統默認 prompt** — 最低

**技能調用流程**：
```
用戶消息
  → 有 1% 可能適用某個技能？
       ├── 是 → Invoke Skill tool → 宣布「使用 [技能] 來 [目的]」
       │         → 有 checklist？→ TodoWrite 每項
       │         → 嚴格按照技能執行
       └── 否 → 回覆（包括澄清問題）
```

### 設計層：為什麼這樣設計？

#### 設計決策 1：「即使 1% 的機率也必須 invoke」——這個規則背後的恐懼是什麼？

**恐懼**：**AI 傾向於跳過紀律，因為它「覺得」自己知道答案。**

這個規則來自一個痛苦的觀察：當 AI 認為「這個問題很簡單」時，它會跳過所有紀律檢查，直接給出答案。但「簡單問題」往往隱藏著未被審視的假設——這些假設在實現階段會變成昂貴的返工。

**1% 規則的真正含義**：不是「1% 機率這個技能有用」，而是「1% 機率你正在理性化跳過紀律」。紅旗表列出的每一個想法（"這只是個簡單問題"、"我可以先看看代碼"）都是 AI 在**說服自己不做應該做的事**。

**為什麼必須是「強制」而不是「建議」**：如果是建議，AI 會每次都選擇「這個情況不需要」。因為跳過紀律的感覺是「節省時間」，但實際代價是「後面花 10 倍時間修復錯誤假設」。只有強制規則能打破這個循環。

**反例**：沒有 1% 規則。用戶說「加個按鈕」，AI 認為太簡單不需要 brainstorming，直接寫了按鈕。結果：按鈕的行為與現有組件衝突、沒有考慮移動端、沒有測試。修復這些問題花了 2 小時，而 brainstorming 只需要 10 分鐘。

---

#### 設計決策 2：技能優先級中「用戶指令 > 技能 > 系統默認」

**決策**：即使用戶指令和技能衝突（用戶說 "don't use TDD"，技能說 "always use TDD"），遵循用戶指令。

**為什麼**：這與 gstack 的 User Sovereignty 原則一致，但 superpowers 的表達方式更直接——「用戶控制一切」。技能是紀律，但紀律是用戶選擇的紀律，不是 AI 強加的紀律。

**反例**：技能覆蓋用戶指令。用戶說「這個項目不需要 TDD」，但 TDD 技能強制寫測試。結果：用戶失去了對自己項目的控制感，開始不信任工具。

## 二、`brainstorming` — 從創意到設計

### 字面層：它做了什麼？

`brainstorming` 是「創意 → 設計 → 規格」的轉換器。它在任何實現工作之前運行，確保「我們在建正確的東西」。

**完整工作流程（9 步，順序強制）：**

```
1. 探索項目上下文
   ├── 讀文件、代碼、最近 commits
   └── 評估範圍：如果太大（多個獨立子系統）→ 先分解

2. 提供視覺伴侶（如果預期有視覺問題）
   └── 必須是獨立消息，不能和澄清問題合併

3. 問澄清問題（一次一個）
   ├── 目的、約束、成功標準
   └── 偏好選擇題，多選題也可以

4. 提出 2-3 個方案
   └── 帶權衡和你的推薦

5. 呈現設計（分段落，每段後獲批准）
   ├── 架構、組件、數據流、錯誤處理、測試
   └── 如果不對 → 回去修改

6. 寫設計文檔
   └── docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md

7. 規格自審
   ├── 佔位符掃描（TBD/TODO）
   ├── 內部一致性
   ├── 範圍檢查
   └── 歧義檢查

8. 用戶審查書面規格
   └── 等待用戶批准 → 如果有修改 → 回到 7

9. 調用 writing-plans 技能
   └── 這是唯一的終端狀態，不能調用其他實現技能
```

**HARD GATE（硬閘門）**：
```
<HARD-GATE>
在呈現設計並獲用戶批准之前，
絕對不能調用任何實現技能、寫任何代碼、
腳手架任何項目、或採取任何實現行動。
這適用於每個項目，無論看起來多簡單。
</HARD-GATE>
```

**反模式**："這太簡單了不需要設計"
- 待辦清單、單一函數工具、配置修改——每個都要走這個流程
- 「簡單」項目是未被審視的假設造成最多浪費工作的地方
- 設計可以很短（真的簡單的項目幾句話），但**必須呈現並獲批准**

**範圍控制機制**：
- 如果請求描述多個獨立子系統（如「做一個有聊天、文件存儲、計費、分析的平臺」）→ 立即標記
- 不要花時間細化一個需要被分解的項目
- 幫助用戶分解：獨立部分是哪些？它們如何關聯？建造順序？
- 每個子項目有自己的 spec → plan → implementation 循環

**設計隔離原則**：
- 把系統拆分成小單元，每個單元有單一明確的目的
- 通過定義良好的接口通信
- 可以獨立理解和測試
- 對於每個單元，你應該能回答：它做什麼？怎麼用它？它依賴什麼？
- 小型、邊界清晰的單元也更容易讓 AI 工作——你在能一次裝進上下文的代碼上推理更好

**Visual Companion 機制**：
- 瀏覽器伴侶是**工具**，不是模式
- 接受伴侶意味著它可用於需要視覺處理的問題，不意味著每個問題都通過瀏覽器
- **每問題決策**：即使用戶接受了，對每個問題決定用瀏覽器還是終端
- 測試：**用戶通過看比通過讀更能理解嗎？**
- 用瀏覽器：mockups、wireframes、佈局比較、架構圖
- 用終端：需求問題、概念選擇、權衡清單、範圍決策

### 設計層：為什麼這樣設計？

#### 設計決策 1：Brainstorming 的「強制結構」——它強制了什麼？為什麼要強制？

**強制了什麼**：
1. **順序強制**：9 個步驟必須按順序完成，不能跳過
2. **批准強制**：每個設計段後必須獲用戶批准
3. **終端狀態強制**：brainstorming 的唯一終端狀態是調用 writing-plans，不能直接跳到實現
4. **「簡單也必須設計」強制**：沒有「這太簡單了」的例外

**為什麼強制**：

AI 的默認行為是「聽到創意 → 開始寫代碼」。這個跳躍太快了——它跳過了「這個创意背後的真實意圖是什麼」「有沒有更好的方式實現這個意圖」「這個创意本身是好的嗎」。

強制結構的作用是把 AI 從「實現模式」拉到「思考模式」。在實現模式下，AI 問「怎麼做」；在思考模式下，AI 問「做什麼」和「為什麼做」。

**反例**：沒有強制結構。用戶說「加個導航欄」，AI 開始寫導航欄代碼。結果：導航欄的交互和現有的路由衝突、沒有考慮登錄/未登錄狀態、沒有處理移動端。返工花了 3 倍時間。如果有 brainstorming，這些問題在寫代碼之前就解決了。

**更深的洞察**：brainstorming 的強制結構是在防止一種特定錯誤——**「正確地實現了錯誤的東西」**。這種錯誤最危險的地方在於：實現本身可能完美，但方向錯了。在 AI 讓實現成本大幅下降的今天，這種錯誤的代價不是「實現太貴」，而是「實現了沒人需要的東西」。

---

#### 設計決策 2：「一次一個問題」的互動模式

**決策**：brainstorming 要求每次只問一個問題，偏好選擇題，避免多選題。

**為什麼**：多個問題同時出現會讓用戶感到被審問，降低回答質量。選擇題比開放問題容易回答因為它縮小了思考範圍——用戶不需要從零構思答案，只需要在已知選項中選擇。

**反例**：「你的目標用戶是誰？他們的核心痛點是什麼？你希望的 MVP 時間線是什麼？你的商業模式是什麼？」——4 個問題，用戶可能只回答前 2 個，後面敷衍。

---

#### 設計決策 3：Visual Companion 的「每問題決策」機制

**決策**：即使用戶接受了視覺伴侶，對每個問題單獨決定用瀏覽器還是終端。

**為什麼**：接受視覺伴侶不等於「所有問題都用瀏覽器」。有些問題本質上是概念的（「個性化在這個產品中意味著什麼？」），用瀏覽器反而低效。有些問題本質上是視覺的（「哪個導航佈局更好？」），用終端無法有效討論。

**反例**：用戶接受了視覺伴侶，然後每個問題都用瀏覽器展示。結果：token 消耗暴增（每個瀏覽器操作需要截圖、DOM 解析），但很多展示只是文字列表，用終端完全可以。

## 三、`writing-plans` — 從設計到實現計劃

### 字面層：它做了什麼？

`writing-plans` 把 brainstorming 產出的設計文檔轉化為**可直接執行的實現計劃**。它假設執行者是「有技能但對這個代碼庫和問題域幾乎零了解的開發者」。

**計劃文檔結構**：

```markdown
# [功能名稱] 實現計劃

> **REQUIRED SUB-SKILL:** Use subagent-driven-development 或 executing-plans

**Goal:** [一句話描述]
**Architecture:** [2-3 句話]
**Tech Stack:** [關鍵技術]

---

### Task N: [組件名稱]

**Files:**
- Create: `exact/path/to/file.py`
- Modify: `exact/path/to/existing.py:123-145`
- Test: `tests/exact/path/to/test.py`

- [ ] **Step 1: 寫失敗的測試**
    [具體代碼]

- [ ] **Step 2: 運行測試確認失敗**
    Run: `pytest ...`
    Expected: FAIL with "..."

- [ ] **Step 3: 寫最小實現代碼**
    [具體代碼]

- [ ] **Step 4: 運行測試確認通過**
    Run: `pytest ...`
    Expected: PASS

- [ ] **Step 5: Commit**
    git add ...
    git commit -m "feat: ..."
```

**關鍵規則**：

1. **No Placeholders（零佔位符）**：
   - 不能有 "TBD"、"TODO"、"稍後實現"
   - 不能有「添加適當的錯誤處理」（不展示具體代碼）
   - 不能有「為以上寫測試」（不展示具體測試代碼）
   - 不能有「類似 Task N」（重複代碼——執行者可能亂序閱讀）
   - 每個步驟必須包含實際內容

2. **Bite-Sized 粒度**：每個步驟是 2-5 分鐘的單一動作
   - 「寫失敗的測試」→ 一步
   - 「運行確認失敗」→ 一步
   - 「寫最小實現代碼」→ 一步
   - 「運行確認通過」→ 一步
   - 「Commit」→ 一步

3. **文件結構先於任務定義**：
   - 在定義任務之前，先映射哪些文件會被創建或修改
   - 每個文件有單一清晰的職責
   - 一起變化的文件應該放在一起（按職責拆分，不是按技術層）

4. **Self-Review（自審）**：
   - Spec coverage：每個規格需求都能指向一個實現任務
   - Placeholder scan：搜索紅旗模式
   - Type consistency：後面任務的類型/簽名/屬性名和前面定義的匹配

### 設計層：為什麼這樣設計？

#### 設計決策 1：計劃文檔格式——為什麼是這個格式？

**為什麼是 TDD 格式（測試 → 失敗 → 實現 → 通過 → Commit）而不是「實現 → 測試」**：

TDD 格式強制執行者在寫代碼之前先思考「這段代碼應該做什麼」。測試是行為規格——它描述了函數的輸入、輸出和約束。先寫測試意味著：
1. 你清楚了這段代碼的預期行為
2. 你有回歸測試證明代碼在未來不會壞
3. 最小實現原則——只寫讓測試通過的代碼，不多寫

**為什麼每步都有具體代碼和命令**：

計劃的消費者是「有技能但對這個代碼庫零了解的開發者」（可能是未來的你，也可能是 subagent）。如果計劃說「添加錯誤處理」，執行者需要自己決定怎麼處理——但這個決定應該在計劃階段做，不是實現階段。

**為什麼不能有佔位符**：

佔位符是「計劃時不想做決定」的表現。但「以後再決定」意味著執行階段遇到問題時要停下來問——這打斷了工作流。計劃的職責是在執行前做所有決定。

**反例**：計劃中有「Task 5: 添加適當的錯誤處理」。執行者讀到這裡，不知道「適當」指的是什麼——是返回 500 錯誤？還是重試？還是降級？回頭問計劃作者，工作流被打斷。

---

#### 設計決策 2：「假設執行者有技能但零上下文」

**決策**：計劃假設執行者是 skilled developer，但對工具鏈和問題域幾乎零了解。

**為什麼**：這個假設精確地描述了 subagent 的狀態——它們有通用的編程能力（知道怎麼寫 Python、怎麼用 git），但不知道這個項目的具體約定（「這個項目用 pytest 不是 unittest」、「錯誤處理用這種模式」）。

計劃需要補足這個差距：給出準確的文件路徑、準確的命令、準確的代碼模式。執行者不需要「怎麼編程」的指導，需要的是「在這個項目裡怎麼做」的指導。

**反例**：假設執行者知道項目約定。計劃說「寫測試」，執行者用了 unittest 但項目用 pytest。返工。

## 四、`executing-plans` — 從計劃到代碼

### 字面層：它做了什麼？

`executing-plans` 是計劃的執行者。它讀取計劃文件，按照步驟執行，在檢查點停下來報告。

**工作流程**：

```
Step 1: 讀取並批判性審查計劃
         → 有疑慮？→ 在開始前向用戶提出
         → 無疑慮？→ 創建 TodoWrite

Step 2: 執行任務
         → 每個任務標記 in_progress
         → 嚴格按照步驟執行
         → 運行驗證
         → 標記 completed

Step 3: 完成開發
         → 所有任務完成後
         → 調用 finishing-a-development-branch
         → 驗證測試、呈現選項、執行選擇
```

**何時停止並求助**：
- 遇到阻塞（缺少依賴、測試失敗、指令不清楚）
- 計劃有阻止開始的关键空白
- 不理解某個指令
- 驗證反覆失敗

**核心規則**：**問而不是猜**。

### 設計層：為什麼這樣設計？

#### 設計決策 1：執行計劃時「遇到阻塞就停止，問而不是猜」

**決策**：executing-plans 在遇到任何阻塞時立即停止，向用戶尋求澄清，而不是嘗試自己解決。

**為什麼**：執行者的職責是「按照計劃做」，不是「修改計劃」。如果計劃有問題，執行者猜測的修復可能不符合計劃作者的意圖。問而不是猜，確保了計劃的權威性和執行的準確性。

**反例**：計劃說「用 X 庫」，但 X 庫沒有安裝。執行者猜測「那就用 Y 庫吧」，安裝了 Y 庫，繼續執行。結果：Y 庫的 API 和 X 不同，後續所有依賴 X API 的步驟都失敗。

## 哲學層：反映了什麼信念？

**信念 1：思考模式和實現模式是兩種完全不同的狀態，需要強制切換。**

Brainstorming 的 HARD GATE、writing-plans 的 No Placeholders 規則、executing-plans 的「問而不是猜」——都在強制 AI 在正確的時刻做正確的事：設計時不想代碼，計劃時不想捷徑，執行時不偏離計劃。

**信念 2：AI 傾向於跳過紀律，因為它「覺得」自己知道答案。**

1% 規則和紅旗表是對這種傾向的直接對抗。AI 的每一個「這很簡單不需要 X」的想法，都是它試圖繞過紀律的信號。superpowers 選擇了「強制」而不是「建議」，因為建議會被 AI 的「自我合理」跳過。

**信念 3：計劃的質量決定了執行的質量。**

writing-plans 的 No Placeholders 規則、Bite-Sized 粒度、具體代碼要求——都在說：計劃不是「大概要做什麼」的清單，而是「具體怎麼做」的腳本。執行者（尤其是 subagent）需要的是腳本，不是清單。

**信念 4：「簡單」項目是最危險的。**

brainstorming 的 Anti-Pattern 明確說：「這太簡單了不需要設計」是最危險的想法。因為簡單項目的假設最容易被忽視——待辦清單、配置修改、單函數工具，這些「簡單」改動的假設錯了，代價和複雜功能一樣大。

## 對 gstack-plus 的啟發

### 啟發 1：「強制 invoke」的精神可以用於 gstack-plus 的模型路由

Superpowers 的 1% 規則說「即使只有 1% 可能適用也必須用」。在 gstack-plus 中，對應的規則可以是：「即使只有 1% 可能這個任務需要 Tier-A 的判斷，也必須路由到 Tier-A」。

**具體應用**：在任務分類器中，設置「保守路由」規則——當分類器不確定任務難度時，默認路由到更高 tier，而不是更低。代價是稍微多一點成本，但避免了「Exec 搞砸了需要從零重來」的巨大代價。

### 啟發 2：Brainstorming 的 HARD GATE 是 gstack-plus 中「規劃和執行分離」的模板

Superpowers 說「在設計批准前絕對不能寫代碼」。gstack-plus 可以說「在 Tier-A 規劃批准前絕對不能讓 Exec 開始工作」。

**具體應用**：在 handoff 模板中加入硬閘門：Exec 模型只能在「規劃已批准 + 任務描述已明確 + 驗證標準已定義」後才接收任務。

### 啟發 3：No Placeholders 規則適用於 Handoff 指令設計

Writing-plans 的 No Placeholders 說「計劃不能有 TBD/TODO」。gstack-plus 的 handoff 指令也應該這樣：給 Exec 模型的指令不能有模糊地帶——「處理錯誤」不夠，要說「如果遇到 NetworkError，重試 2 次，然後返回 500」。

### 啟發 4：「問而不是猜」是 Exec 模型失敗時的良好行為模式

Executing-plans 遇到阻塞就停下來問。gstack-plus 的 Exec 模型也應該這樣：遇到不確定的情況時，返回給 Claude 詢問，而不是自行猜測。

### 啟發 5：Bite-Sized 粒度適用於 Exec 模型的任務拆分

Writing-plans 要求每個步驟是 2-5 分鐘的單一動作。gstack-plus 給 Exec 模型的任務也應該這樣拆分：不是「做整個 billing 模塊」，而是「寫 Payment.process() 函數，接受 amount 和 currency 參數，返回 transaction_id」。

**在 gstack-plus 中的具體應用——禁用詞清單**：

給 Qwen Exec 的任務描述中，以下詞語出現時必須在派發前替換：

| 禁用詞 | 問題 | 替換方式 |
|-------|------|---------|
| 「適當的」 | Qwen 會用自己的標準判斷「適當」 | 改為具體條件：「不超過 100ms 的」 |
| 「合理的」 | 同上 | 改為可測量標準：「通過 lint 規則的」 |
| 「如果需要」 | Qwen 通常會選擇「不需要」 | 改為明確指令：「必須添加」或「不需要添加」 |
| 「根據情況」 | 把決策推給 Qwen，但 Qwen 沒有全局視角 | 在派發前由 Claude 做決策，明確寫入 |
| 「類似地」「以此類推」 | Qwen 可能推斷錯誤的類比對象 | 明確列出所有需要修改的位置 |

❌ 錯誤的任務描述：「在適當的位置添加錯誤日誌」
✅ 正確的任務描述：「在 src/api/user.ts 的第 45 行 catch 塊中，添加 console.error，
   格式為：`[ERROR][${new Date().toISOString()}][fetchUser] ${error.message}`」

## Superpowers vs gstack 規劃技能的關鍵差異

| 維度 | gstack | Superpowers |
|------|--------|-------------|
| 規劃觸發 | 用戶調用 `/plan-*` 技能 | 強制 brainstorming 在任何實現前 |
| 規劃角色 | 不同角色有不同視角（CEO/Eng/Design） | 統一的「產品設計師 + 架構師」角色 |
| 規劃輸出 | 設計文檔 + review findings | spec 文檔 + 實現計劃 |
| 強制性 | 可選（但 `/ship` 會建議） | 強制（HARD GATE） |
| 用戶互動 | AskUserQuestion 在關鍵節點 | 每次一個問題，逐步批准 |
| 計劃格式 | 自然語言 + ASCII 圖 | TDD 格式 + 具體代碼 + 具體命令 |
| 粒度 | 功能級 | 步驟級（2-5 分鐘） |

**核心洞察**：gstack 的規劃是「多角度審查」，superpowers 的規劃是「從創意到腳本」。gstack 假設用戶已經有想法，需要確保想法是好的；superpowers 假設用戶只有創意，需要先把創意變成可執行的腳本。
